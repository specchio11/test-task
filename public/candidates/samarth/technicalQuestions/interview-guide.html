📖 追问 1 面试官参考答案 / Follow-up 1 Interviewer Reference

✅ Context 适合的场景 / Where Context Fits

  • 状态层级浅、变更频率低（user session、theme、locale、auth token）
    Shallow state, low change frequency (user session, theme, locale, auth token)
  • 项目规模小到中等，不需要 time-travel debugging / 复杂 middleware
    Small-to-medium project, no need for time-travel or complex middleware
  • 不引入额外依赖，对 bundle size 友好
    No extra dependency, friendly to bundle size

⚠️ Context 的代价 / Context Costs (a senior should mention)

  • 重渲染陷阱：provider value 里任何一个字段变了，所有 consume 这个 Context 的组件都会重新渲染
    Any change in the provider value re-renders every consumer
  • 缓解手段：
    • 把 Context 拆细（user / theme / auth 各自独立 provider）
    • provider value 用 useMemo 稳定引用
    • 需要细粒度订阅时换 Zustand / Jotai
  • 不适合放 server state：cache、refetch、stale-while-revalidate、retry 这些用 Context 手写极难维护，应该用 React Query / SWR
    Server state belongs in React Query / SWR, not Context

📊 备选方案对比 / Alternatives Trade-off

方案适合不适合
Context + hooks少量浅层共享 state状态多、变更频繁、跨组件细粒度订阅
Zustand中等复杂度、想要 selector 细粒度订阅、minimal API需要严格的 reducer + middleware 标准化
Jotai原子化 state，本地推导 derived state 多不熟悉 atom 心智模型的团队
Redux Toolkit大团队、严格规范、time-travel debug小项目过重
React Query / SWR所有 server state（cache、refetch、invalidation）纯 client state

▸ Sub-Q1 — 实时数据流架构 / Real-time Data Stream Architecture

🎤 提问 / Ask · Sub-Q1
"If your dashboard had live data — say streaming price ticks, or auto-refreshed positions — how did you handle the streaming side? Specifically: what does your reconnect look like, do you backfill missed ticks when you reconnect, and what happens when the user backgrounds the tab for a while?"
"如果你们 dashboard 有实时数据——比如行情推送，或者持仓自动刷新——你是怎么处理流式那一层的？具体说：你们的断线重连长什么样？连上之后会不会补齐这段时间漏掉的 tick？用户把 tab 切到后台一段时间再回来，又会怎么处理？"
📖 Sub-Q1 参考答案 / Reference

🔌 传输协议选型 / Transport choice

  • WebSocket — 双向、低延迟，行情/聊天/协作的标配
  • SSE (Server-Sent Events) — 单向 server→client，比 WS 简单，HTTP/2 友好，自带断线重连
  • Long polling — 兜底方案；现代项目几乎不再用

🔁 断线重连 / Reconnection strategy

  • 🟢 指数退避 + jitter（1s → 2s → 4s → 8s + 随机偏移），避免雪崩重连
    Exponential backoff with jitter to avoid thundering-herd reconnects
  • 🟢 区分错误类型：网络断 vs auth 失效 vs 服务端 5xx → 重连策略不同
    Distinguish network drop vs auth expiry vs 5xx — different reconnect strategy
  • 🟡 最大重连次数 + 用户提示，长时间断要降级 UI（show stale banner）
    Max retries + degraded UI banner ("data may be stale")

⏪ Backfill / 补齐漏 tick

  • 🟢 重连之后用 REST 拉 [lastSeenTs, now] 这段历史，合并进 buffer，再 resume 流
    On reconnect: REST-fetch [lastSeenTs, now] gap, merge, then resume stream
  • 🟢 去重：每条消息带 sequence number 或 timestamp；merge 时按 seq 去重
    Dedupe via sequence number or timestamp on merge
  • 🟢 乱序处理：late tick 按 seq 插回正确位置，或丢弃（依赖业务容忍度）
    Out-of-order ticks: reorder by seq or drop based on tolerance

🌒 Tab 后台 / Page visibility

  • 🟢 Page Visibility API 监听 visibilitychange
  • 选择策略（两种都合理）：
    (a) 暂停 / throttle 渲染但保持 WS 连接 → 切回来重新 render 最新 snapshot
    (b) 断开 WS 节省资源 → 切回来走 reconnect + backfill 流程
  • 🚩 红旗：tab 后台时仍然 60 FPS 渲染，电量爆炸

💾 内存 / 缓冲管理

  • 🟢 Ring buffer 或 max-N 限制——长时间运行的 dashboard，无上限 buffer 必 OOM
    Ring buffer / capped buffer — unbounded buffer = OOM in long-running dashboards
  • 🟢 组件 unmount 时 cleanup：close WS、clear interval、abort fetch（典型 RN 多页面切换坑）
    Cleanup on unmount — close WS / clear intervals / abort fetches

📈 业务层细节 / Domain edge cases (金融特有)

  • 市场闭市 gap：周末 / 收盘后没 tick — 显示 null gap / interpolation / 还是 connect-the-dots？通常显示 break + 闭市 banner
  • 停牌 / halt：tick 突然消失但又不是断网，需要服务端推 halt 信号
  • 时区：tick 时间戳是 exchange local（NYSE = ET）还是 UTC？聚合 OHLC 时按哪个时区切日界？
🎯 Sub-Q1 候选人回答评价

🟢 强 / Strong:

  • 主动说出 reconnect + backfill + dedupe 三件套
    Names reconnect + backfill + dedupe unprompted
  • 提到 Page Visibility API / tab 后台的处理决策
    Mentions Page Visibility / tab background handling
  • 主动提到 buffer cap / memory 管理
    Unprompted on buffer cap / memory
  • 能聊业务层 edge case（闭市 gap、stale data UI 提示）
    Talks about domain edge cases (market close gap, stale banner)

🟡 中等 / Okay:

  • 知道要重连但没想过 backfill；模糊提到"会处理乱序但记不清细节"
    Knows about reconnect but no backfill plan; vague on ordering

🔴 弱 / Weak:

  • "用 socket.io 自带就行了" — "socket.io handles it for us"
    → 🚩 socket.io 的 reconnect 不包括 backfill / dedupe / 业务层 gap，这答案说明没真处理过线上 incident
  • "buffer 一直涨没问题，浏览器有 GC" — "Buffer grows fine, browser will GC"
    → 🚩 GC 不会回收你 ref 着的数组，纯概念错误

"On the interactivity side — did users need to zoom or pan the chart? Were there multiple charts that needed to sync on hover (e.g. hover on the price chart and the volume chart shows the same crosshair)? And if it's mobile, what about pinch and two-finger pan? How did you actually implement that?"

📖 Sub-Q2 参考答案 / Reference

🔍 Zoom on large data

  • 🟢 多分辨率预聚合：服务端按 zoom level 预生成 1m / 5m / 1h / 1d bars，前端按当前 viewport 拉对应粒度（典型 K 线方案）
    Multi-resolution pre-aggregation server-side; client requests the resolution that fits the viewport
  • 🟡 客户端 re-downsample：zoom 时用 LTTB 重新算（适合数据量不大、不想加 server 复杂度）
    Client-side re-downsample via LTTB — simpler but heavier client compute
  • 🟢 Viewport-based fetching：pan 时按时间范围 lazy fetch，类似无限滚动列表
    Viewport-based / range-request fetching as user pans

🔗 跨图同步 / Cross-chart sync

  • 🟢 共享 hover state — Context / Zustand / Jotai 存当前 hover 的时间戳，所有 chart 订阅并画 crosshair
    Shared hover state in store; each chart subscribes and renders its own crosshair
  • 🟢 事件总线 — 一个 chart hover 时 dispatch event，其他 chart 监听（更松耦合）
    Event bus — looser coupling than shared store
  • 🟡 chart lib 自带 — Highcharts 有 Highcharts.Pointer.onContainerMouseMove + syncCharts 范例；Recharts / Chart.js 通常要自己接线
    Some libs have built-in sync (Highcharts demo); most need wiring
  • 性能要点：hover state 更新要 throttle 到 ~30 FPS，否则 mousemove 一秒触发 ~60 次 → 多 chart 都重渲

📱 Mobile 手势 / Mobile gestures (RN 强相关)

  • Web：用 Pointer Events（统一 mouse/touch/pen），或 PointerEvent + Pointer Lock；pinch 通过 2 个 active pointer 算距离差
  • React Native：react-native-gesture-handler + react-native-reanimated 的 worklet — 手势处理跑在 UI thread 上，不经过 JS bridge，60 FPS 流畅
    RN: gesture-handler + Reanimated worklets — gesture runs on UI thread, no JS bridge
  • Pinch zoom：scale 应用到 chart 的 x-axis domain，而不是 CSS transform（CSS transform 会糊）
    Pinch → x-axis domain change, NOT CSS transform (transform = blurry)
  • 双指 pan 和 单指 scroll 要区分（典型坑：单指 pan chart 会和页面 scroll 冲突，要 preventDefault / touch-action: none）

🪟 Brush / Range selection

  • 用户拖一个区间 → main chart 缩放到该区间 + fetch 该区间高精度数据
  • 典型 UI：上方一个 overview chart（全量低精度）+ 下方 detail chart（区间高精度）

🚫 反模式 / Anti-patterns

  • 🚩 zoom 时不重新降采样，仍然渲染 50k 个 SVG 节点
  • 🚩 多图同步用 setState 在 mousemove 里直接调，没 throttle → 一秒 60 次 reconcile
  • 🚩 mobile pinch 用 CSS transform scale → 文字 / 线条糊掉，axis label 不更新
  • 🚩 没处理 page scroll 冲突，用户在 chart 上 pan 结果页面跟着滚

📚 Step 1 — 常见图表库 landscape / Library Landscape

库渲染层核心定位适合场景 / 陷阱

🚀 Step 2 — 大数据量优化 / Large-Dataset Techniques

  • 🟢 数据降采样 (Downsampling) — 经典算法 LTTB (Largest Triangle Three Buckets) 在保留视觉趋势的同时把点数从 N→K（如 50k→500）
    Downsampling via LTTB preserves visual trend while collapsing N→K
  • 🟢 服务端聚合 — 只传当前 zoom level 需要的精度（K 线每个时间窗口给 OHLC 而不是每 tick）
    Server-side aggregation — only ship the precision needed for current zoom level (OHLC per window vs every tick)
  • 🟢 用 Canvas / WebGL 而非 SVG — SVG 一个点 = 一个 DOM 节点，>5k 点必卡
    Canvas/WebGL over SVG — each SVG node is a DOM node, >5k points = slow
  • 🟡 Web Worker 预处理 — 把数据清洗 / 降采样移出主线程
    Web Worker for preprocessing — move cleaning/downsampling off the main thread
  • 🟡 Progressive rendering — 大数据分批 / 异步渲染，避免一次性 block UI
    Render in chunks via requestIdleCallback to avoid blocking UI

⚡ Step 2 — 频繁更新 / 动画性能 / Update & Animation Perf

  • 🟢 用 chart.update() / partial update API，不要 unmount + remount 整个 chart 实例（最常见的性能事故）
    Use the chart's update API; do NOT unmount+remount on data change (most common perf bug)
  • 🟢 memoize options 对象 — Recharts / Highcharts 的 options 是引用比较，每次新对象 = 全量重渲
    Memoize the options object — most libs reference-compare; new object = full re-render
  • 🟢 实时流要 throttle / debounce — WebSocket 行情 60+ updates/sec，必须节流到 ~10 FPS 渲染
    Throttle realtime streams — 60+ updates/sec must be throttled to ~10 FPS
  • 🟢 更新时关掉 animation — animation 只在首次挂载时开，data update 时禁用（否则插值动画会和新数据打架）
    Disable animation on update — keep it for initial mount only, otherwise interpolation fights new data
  • 🟡 React.memo + 自定义 comparator — 父组件 re-render 时阻止 chart 重渲
    React.memo + custom comparator — prevent re-render when parent re-renders
  • 🟡 requestAnimationFrame 批量化 — 把多个 quick updates 合并到下一帧
    Batch updates via requestAnimationFrame

🚫 典型反模式 / Anti-patterns

  • 🚩 每次 data update 都 new Chart() 或 unmount&remount → 全量重建 DOM + 失去动画
  • 🚩 直接把 5k+ raw points 塞 Recharts → SVG DOM 节点爆炸
  • 🚩 行情流 setState 不做 throttle → React 一秒 reconcile 60 次
  • 🚩 把 chart options 写在 JSX 内联 → 每次 render 都是新引用，全量重画

📖 Sub-Q2 参考答案 / Reference Answer

常见通信模式 / Common patterns (任选其一或组合):

  • 🟢 共享 store（Module Federation 暴露 Redux / Zustand instance）— 最强一致性，但耦合度高

  • 🟢 Custom Events / window.dispatchEvent — 松耦合，最常见；定义清晰的 event payload schema 是关键

  • 🟡 Event bus / pub-sub 库（如 mitt）— 比裸 window event 更可测、可类型化

  • 🟡 URL / query params — 最弱耦合，但只能传简单原语
    URL/query params — weakest coupling, primitives only
  • 🟠 postMessage — iframe 方案唯一选项；需要 serialization，开销较大
    postMessage — only option for iframes; serialization overhead

📖 追问 3 面试官参考答案 / Follow-up 3 Interviewer Reference

✅ 微前端真正的适用场景 / When Micro-frontends Actually Help

  • 多团队并行开发同一个 web app，且每个团队想独立部署（避免大一统 monorepo build）
    Multiple teams shipping into one app, each wanting independent deploys
  • 需要技术栈隔离（一部分用 React，一部分迁移中的 legacy Angular/Vue）
    Tech stack isolation (e.g. React + legacy Angular co-existing)
  • 典型规模：50+ 工程师 / 多产品线 / 多 BU
    Typical scale: 50+ engineers, multi-product, multi-BU

⚠️ 微前端的代价 / Micro-frontend Costs

  • 共享依赖（React 版本、router 版本）需要协调，否则 bundle 会重复且行为冲突
    Shared deps need coordination, otherwise bundle bloat + version conflicts
  • 跨 micro-app 的状态共享困难（不能直接 prop drill）
    Cross-app state sharing is hard (can't just prop-drill)
  • 整体加载时间和构建管线复杂度大幅上升
    Total load time and pipeline complexity jump significantly
  • Design system 和组件库必须严格集中治理
    Design system and component library must be strictly centralized

🛠️ 常见技术方案 / Common Approaches

  • Webpack Module Federation：现在最主流，运行时动态加载 remote bundle
  • single-spa：早期方案，把多个 SPA 挂载到同一个 shell
  • iframe：最强隔离但 UX 最差（跨 iframe 通信 / 滚动条 / 弹窗都难处理）
  • Build-time composition（npm packages + monorepo）：本质不是微前端，是模块化

🚫 移动端微前端 / Mobile Micro-frontends?

  • RN 几乎没有真正的"微前端"——可以做 code splitting / dynamic bundles（Re.Pack、Repack），但概念和 web 微前端差距很大
    RN doesn't really have micro-frontends; closest is dynamic bundles (Re.Pack)
  • Copi 这种规模的金融 app 应用微前端 = over-engineering
    A Copi-scale fintech app applying micro-FE = over-engineering
