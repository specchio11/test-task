🎯 主问题 / Main Question · 🌐 Ownership 边界 / Scope of Ownership ~2 min

🎤 提问 / Ask
"Your resume says you owned the frontend architecture for the Copi mobile app. Before we go into the design itself — can you tell me what 'owned' means in your case? How big was the frontend team, and which calls were yours to make?"
"简历上写你 own 了 Copi 移动端的前端架构。在聊设计之前，先想搞清楚 ownership 的边界——前端一共几个人？哪些决定是你拍板的？"
⚠️ 面试官提示 — 这一问的目的是先校准"own"的真实含义。如果他说"前端就我一个"，那后面追问的答案可以完全代表他个人；如果他说"我们有 2-3 人，我 lead"，要追问"另外几位的 level 是什么、谁拍板架构 decision"。
Interviewer note — Calibrate what "own" means before the deep dive. If he was the sole frontend, the follow-up answers fully represent him. If he led 2-3 people, follow up on the others' seniority and who made the architecture calls.
📖 面试官参考答案 / Interviewer Reference — Q1 Ownership

Copi 背景假设 / Background assumption：Copi 是 Vancouver 的小型 fintech 创业公司，按 7 个月工作期 + 简历"owned"措辞推断，大概率前端就 1 人（他自己），或 1-2 人小组。

合理的 ownership 维度 / Reasonable scope a real owner should cover:

  • 🟢 Tech stack 选型：选 RN + Expo 而不是 Flutter / 原生 / Web 的理由
    Tech stack choice — RN+Expo vs Flutter/native/web and why
  • 🟢 架构 layer 划分：navigation、state、API client、UI 组件、theme 等谁先谁后
    Layering — navigation, state, API client, UI primitives, theme
  • 🟢 跨端协作契约：和 backend 怎么对 API schema、和 design 怎么对组件库
    Cross-team contracts — API schema with backend, component library with design
  • 🟡 发布管线：EAS Build / OTA update / app store 流程
    Release pipeline — EAS Build / OTA / app store
  • 🟡 Code review 制度：自己提的 PR 谁来 review？是否有 staff/lead 在上面把关
    Code review — who reviews his PRs? Is there a senior gatekeeper?
🎯 候选人回答评价 / Candidate Answer Evaluation — Q1

🟢 强回答 / Strong answers:

  • 诚实给出团队规模（"前端就我一个" 或 "我加一个 junior"），并清晰列出 哪些决定是自己拍板、哪些和别人 align 后定
    Honest team size + clear list of which decisions were his vs co-decided
  • 主动 frame 出 "我也要和 backend 对 API 契约 / 和 design 对组件库" 这类边界 contract 意识
    Frames cross-team contracts unprompted (a senior signal)

🟡 中等回答 / Okay answers:

  • 给出了团队规模但说不清自己的决定边界 — Gives team size but vague on his own decision boundary
    → 追问具体一个 decision example（比如"上 Expo 是谁决定的？"）/ Probe with a concrete decision example

🔴 弱回答 / Weak answers:

  • "整个团队都是 owner / We all owned it together" — "We all owned it together"
    → 🚩 含糊推卸，可能没真正主导 / Vague hedging, possibly not the real driver
  • 列了一堆用了什么（React Native、Expo、TypeScript）但答不出谁定的 — Lists tech used but can't say who picked it
    → 🚩 user 视角而非 owner 视角 / User perspective, not owner perspective
➜ 追问 1 / Follow-up 1 · 🎯 State 选型 & 为什么 / State Choice & Rationale ~4 min CORE

📌 关联背景 / Background: 简历明确写 "state management (Context + custom hooks)"。这一问分两步——先让他 map 出 state 管理的 landscape（看眼界），再问 Copi 为什么选了 Context（看 reasoning）。
Resume specifies "Context + custom hooks". This probe runs in two steps: first map the landscape (breadth), then ask why Context for Copi (reasoning).

🎤 提问 / Ask（两步走 / two steps）

先问第一步，等他答完再问第二步。
Ask step 1 first, wait for the answer, then ask step 2.

▸ 第一步 / Step 1 — Landscape

"Let's stay on state management for a bit. First — can you walk me through the main state-management options you know? What's actually out there in the React ecosystem?"
"我们在 state 管理这块停留一下。先问你——你了解的 state 管理方案都有哪几种？React 生态里都有什么？"

▸ 第二步 / Step 2 — Choice

"Got it. So why did you go with Context plus custom hooks for Copi? What made it the right fit over the others you just mentioned?"
"嗯。那当时 Copi 这个项目，你为什么选了 Context + custom hooks？跟你刚说的那几个比起来，为什么 Context 在这里更合适？"
⚠️ 面试官提示 — 第一步本身就是信号。如果他只能说出 1-2 个方案（比如只知道 Redux），眼界就已经定级了；这种情况下第二步的 "why" 答案 weight 会下降。如果他能流利列出 4-5 种（Context / Redux / Zustand / Jotai / Recoil）并主动区分出 React Query / SWR 这类 server-state 工具，第二步要重点听 trade-off reasoning。
Interviewer note — Step 1 is already a signal. If he names only 1-2 options (e.g. just Redux), his breadth is already calibrated and step 2's "why" carries less weight. If he fluently lists 4-5 (Context / Redux / Zustand / Jotai / Recoil) and separately surfaces React Query/SWR for server state, focus step 2 on his trade-off reasoning.

"提示一下——Context 在一个 provider 里塞太多东西时有个著名的重渲染陷阱。常见的绕开方法是什么？项目到什么规模会真的开始痛？"
📖 追问 1 面试官参考答案 / Follow-up 1 Interviewer Reference

✅ Context 适合的场景 / Where Context Fits

  • 状态层级浅、变更频率低（user session、theme、locale、auth token）
    Shallow state, low change frequency (user session, theme, locale, auth token)
  • 项目规模小到中等，不需要 time-travel debugging / 复杂 middleware
    Small-to-medium project, no need for time-travel or complex middleware
  • 不引入额外依赖，对 bundle size 友好
    No extra dependency, friendly to bundle size

⚠️ Context 的代价 / Context Costs (a senior should mention)

  • 重渲染陷阱：provider value 里任何一个字段变了，所有 consume 这个 Context 的组件都会重新渲染
    Any change in the provider value re-renders every consumer
  • 缓解手段：
    • 把 Context 拆细（user / theme / auth 各自独立 provider）
    • provider value 用 useMemo 稳定引用
    • 需要细粒度订阅时换 Zustand / Jotai
  • 不适合放 server state：cache、refetch、stale-while-revalidate、retry 这些用 Context 手写极难维护，应该用 React Query / SWR
    Server state belongs in React Query / SWR, not Context

📊 备选方案对比 / Alternatives Trade-off

方案适合不适合
Context + hooks少量浅层共享 state状态多、变更频繁、跨组件细粒度订阅
Zustand中等复杂度、想要 selector 细粒度订阅、minimal API需要严格的 reducer + middleware 标准化
Jotai原子化 state，本地推导 derived state 多不熟悉 atom 心智模型的团队
Redux Toolkit大团队、严格规范、time-travel debug小项目过重
React Query / SWR所有 server state（cache、refetch、invalidation）纯 client state
🎯 追问 1 候选人回答评价 / Follow-up 1 Candidate Answer Evaluation

🟢 强回答 / Strong answers:

  • 主动提到 Context 的 re-render 陷阱，并说明用 provider 拆细 + useMemo 或换 Zustand 的迁移条件
    Names the re-render trap and the mitigation (split providers + useMemo, or migrate to Zustand)
  • 明确把 server state 单独处理（"账户和交易我用 React Query / SWR，Context 只放 auth + theme"）
    Treats server state separately (React Query/SWR), Context only for auth + theme
  • 能给出项目规模阈值（"如果 state 变成 20+ 个域，我会迁移到 Zustand"）
    Names a scale threshold for migration

🟡 中等回答 / Okay answers:

  • 知道有 Zustand / Redux 但说不清取舍 — Knows alternatives exist but vague on trade-offs
    → 追问 "Redux 你最后没选是因为什么？" / Probe why Redux was rejected
  • "Context 简单不用引入依赖" 是合理的但 surface 的回答 — "Context is simpler, no extra dep" is reasonable but shallow
    → 追问 "项目变大什么信号会让你换？" / Probe the migration signal

🔴 弱回答 / Weak answers:

  • 没听说过 re-render 陷阱 — Hasn't heard of the re-render trap
    → 🚩 自陈"owned 架构"但对最基本的性能特性不了解 / Claims to own arch but doesn't know its core perf characteristic
  • 所有 state（包括 transaction list）都塞 Context — All state including transaction list goes in Context
    → 🚩 cache 失效 / refetch 全部自己手写，工程债大 / Will hand-roll cache & refetch — big engineering debt
  • "Redux 太复杂所以选 Context" 没有真正比较过 — "Redux is too complex" without ever trying it
    → 🚩 决策靠传闻而非评估 / Decided on hearsay, not evaluation
➜ 追问 2 / Follow-up 2 · 🎯 微前端接触 / Micro-frontend Exposure ~2 min STRETCH

📌 关联背景 / Background: Copi 移动端不太可能用微前端（项目规模太小），所以这是一个 stretch 题——目的是看候选人有没有意识到"微前端是大组织才需要的工具"、能否说出适用场景与代价。
A Copi-scale mobile app almost never needs micro-frontends. Stretch probe: does he recognize that micro-FE is a tool for large orgs, and can he articulate use cases + costs?

🎤 提问 / Ask
"Switching gears for a second — have you ever worked with a micro-frontend setup? Anywhere, not just at Copi."
"换个话题问一下——你接触过微前端架构吗？任何项目都行，不限 Copi。"
⚠️ 面试官提示 — 如果他说"用过"，立刻深问具体方案（Module Federation / single-spa / iframe）和团队规模。如果他说"没接触过"，不要纠缠——让他口头推理"什么场景才需要微前端"，看他能不能 reason 出来。
Interviewer note — If he says yes, drill into the specific approach (Module Federation / single-spa / iframe) and team size. If no, don't grind — ask him to reason out when micro-FE would be needed.
📖 追问 2 面试官参考答案 / Follow-up 2 Interviewer Reference

✅ 微前端真正的适用场景 / When Micro-frontends Actually Help

  • 多团队并行开发同一个 web app，且每个团队想独立部署（避免大一统 monorepo build）
    Multiple teams shipping into one app, each wanting independent deploys
  • 需要技术栈隔离（一部分用 React，一部分迁移中的 legacy Angular/Vue）
    Tech stack isolation (e.g. React + legacy Angular co-existing)
  • 典型规模：50+ 工程师 / 多产品线 / 多 BU
    Typical scale: 50+ engineers, multi-product, multi-BU

⚠️ 微前端的代价 / Micro-frontend Costs

  • 共享依赖（React 版本、router 版本）需要协调，否则 bundle 会重复且行为冲突
    Shared deps need coordination, otherwise bundle bloat + version conflicts
  • 跨 micro-app 的状态共享困难（不能直接 prop drill）
    Cross-app state sharing is hard (can't just prop-drill)
  • 整体加载时间和构建管线复杂度大幅上升
    Total load time and pipeline complexity jump significantly
  • Design system 和组件库必须严格集中治理
    Design system and component library must be strictly centralized

🛠️ 常见技术方案 / Common Approaches

  • Webpack Module Federation：现在最主流，运行时动态加载 remote bundle
  • single-spa：早期方案，把多个 SPA 挂载到同一个 shell
  • iframe：最强隔离但 UX 最差（跨 iframe 通信 / 滚动条 / 弹窗都难处理）
  • Build-time composition（npm packages + monorepo）：本质不是微前端，是模块化

🚫 移动端微前端 / Mobile Micro-frontends?

  • RN 几乎没有真正的"微前端"——可以做 code splitting / dynamic bundles（Re.Pack、Repack），但概念和 web 微前端差距很大
    RN doesn't really have micro-frontends; closest is dynamic bundles (Re.Pack)
  • Copi 这种规模的金融 app 应用微前端 = over-engineering
    A Copi-scale fintech app applying micro-FE = over-engineering
🎯 追问 2 候选人回答评价 / Follow-up 2 Candidate Answer Evaluation

🟢 强回答 / Strong answers:

  • 真用过 Module Federation / single-spa，并能说出最大的痛点（shared deps / 跨 app state）
    Has used Module Federation/single-spa and names a real pain point
  • 没用过但能清晰 reason 出"什么规模、什么组织结构才需要"——明确指出 Copi 不需要、移动端基本无相关方案
    Hasn't used it but reasons clearly about when it's warranted, recognizes Copi-scale doesn't need it

🟡 中等回答 / Okay answers:

  • 知道概念能描述但没用过，给不出代价分析 — Knows the concept but no cost-side analysis
    → 中性，不扣分 / Neutral, no penalty

🔴 弱回答 / Weak answers:

  • "微前端就是把前端拆成多个仓库" — "Micro-FE is just splitting into multiple repos"
    → 🚩 把 monorepo / 仓库拆分和微前端混淆 / Confuses repo split with micro-FE
  • "完全没听说过，那是什么" — "Never heard of it"
    → ⚠️ 对自陈 senior frontend 来说有点出戏，但不是 deal-breaker（很多 RN 工程师确实不接触）/ Mildly off-narrative for a self-described senior FE; not a deal-breaker
