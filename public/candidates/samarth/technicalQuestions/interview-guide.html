📖 追问 1 面试官参考答案 / Follow-up 1 Interviewer Reference

✅ Context 适合的场景 / Where Context Fits

  • 状态层级浅、变更频率低（user session、theme、locale、auth token）
    Shallow state, low change frequency (user session, theme, locale, auth token)
  • 项目规模小到中等，不需要 time-travel debugging / 复杂 middleware
    Small-to-medium project, no need for time-travel or complex middleware
  • 不引入额外依赖，对 bundle size 友好
    No extra dependency, friendly to bundle size

⚠️ Context 的代价 / Context Costs (a senior should mention)

  • 重渲染陷阱：provider value 里任何一个字段变了，所有 consume 这个 Context 的组件都会重新渲染
    Any change in the provider value re-renders every consumer
  • 缓解手段：
    • 把 Context 拆细（user / theme / auth 各自独立 provider）
    • provider value 用 useMemo 稳定引用
    • 需要细粒度订阅时换 Zustand / Jotai
  • 不适合放 server state：cache、refetch、stale-while-revalidate、retry 这些用 Context 手写极难维护，应该用 React Query / SWR
    Server state belongs in React Query / SWR, not Context

📊 备选方案对比 / Alternatives Trade-off

方案适合不适合
Context + hooks少量浅层共享 state状态多、变更频繁、跨组件细粒度订阅
Zustand中等复杂度、想要 selector 细粒度订阅、minimal API需要严格的 reducer + middleware 标准化
Jotai原子化 state，本地推导 derived state 多不熟悉 atom 心智模型的团队
Redux Toolkit大团队、严格规范、time-travel debug小项目过重
React Query / SWR所有 server state（cache、refetch、invalidation）纯 client state

▸ Sub-Q1 — 实时数据流架构 / Real-time Data Stream Architecture

🎤 提问 / Ask · Sub-Q1
"If your dashboard had live data — say streaming price ticks, or auto-refreshed positions — how did you handle the streaming side? Specifically: what does your reconnect look like, do you backfill missed ticks when you reconnect, and what happens when the user backgrounds the tab for a while?"
"如果你们 dashboard 有实时数据——比如行情推送，或者持仓自动刷新——你是怎么处理流式那一层的？具体说：你们的断线重连长什么样？连上之后会不会补齐这段时间漏掉的 tick？用户把 tab 切到后台一段时间再回来，又会怎么处理？"
📖 Sub-Q1 参考答案 / Reference

🔌 传输协议选型 / Transport choice

  • WebSocket — 双向、低延迟，行情/聊天/协作的标配
  • SSE (Server-Sent Events) — 单向 server→client，比 WS 简单，HTTP/2 友好，自带断线重连
  • Long polling — 兜底方案；现代项目几乎不再用

🔁 断线重连 / Reconnection strategy

  • 🟢 指数退避 + jitter（1s → 2s → 4s → 8s + 随机偏移），避免雪崩重连
    Exponential backoff with jitter to avoid thundering-herd reconnects
  • 🟢 区分错误类型：网络断 vs auth 失效 vs 服务端 5xx → 重连策略不同
    Distinguish network drop vs auth expiry vs 5xx — different reconnect strategy
  • 🟡 最大重连次数 + 用户提示，长时间断要降级 UI（show stale banner）
    Max retries + degraded UI banner ("data may be stale")

⏪ Backfill / 补齐漏 tick

  • 🟢 重连之后用 REST 拉 [lastSeenTs, now] 这段历史，合并进 buffer，再 resume 流
    On reconnect: REST-fetch [lastSeenTs, now] gap, merge, then resume stream
  • 🟢 去重：每条消息带 sequence number 或 timestamp；merge 时按 seq 去重
    Dedupe via sequence number or timestamp on merge
  • 🟢 乱序处理：late tick 按 seq 插回正确位置，或丢弃（依赖业务容忍度）
    Out-of-order ticks: reorder by seq or drop based on tolerance

🌒 Tab 后台 / Page visibility

  • 🟢 Page Visibility API 监听 visibilitychange
  • 选择策略（两种都合理）：
    (a) 暂停 / throttle 渲染但保持 WS 连接 → 切回来重新 render 最新 snapshot
    (b) 断开 WS 节省资源 → 切回来走 reconnect + backfill 流程
  • 🚩 红旗：tab 后台时仍然 60 FPS 渲染，电量爆炸

💾 内存 / 缓冲管理

  • 🟢 Ring buffer 或 max-N 限制——长时间运行的 dashboard，无上限 buffer 必 OOM
    Ring buffer / capped buffer — unbounded buffer = OOM in long-running dashboards
  • 🟢 组件 unmount 时 cleanup：close WS、clear interval、abort fetch（典型 RN 多页面切换坑）
    Cleanup on unmount — close WS / clear intervals / abort fetches

📈 业务层细节 / Domain edge cases (金融特有)

  • 市场闭市 gap：周末 / 收盘后没 tick — 显示 null gap / interpolation / 还是 connect-the-dots？通常显示 break + 闭市 banner
  • 停牌 / halt：tick 突然消失但又不是断网，需要服务端推 halt 信号
  • 时区：tick 时间戳是 exchange local（NYSE = ET）还是 UTC？聚合 OHLC 时按哪个时区切日界？

📖 Sub-Q2 参考答案 / Reference

🔍 Zoom on large data

  • 🟢 多分辨率预聚合：服务端按 zoom level 预生成 1m / 5m / 1h / 1d bars，前端按当前 viewport 拉对应粒度（典型 K 线方案）
    Multi-resolution pre-aggregation server-side; client requests the resolution that fits the viewport
  • 🟡 客户端 re-downsample：zoom 时用 LTTB 重新算（适合数据量不大、不想加 server 复杂度）
    Client-side re-downsample via LTTB — simpler but heavier client compute
  • 🟢 Viewport-based fetching：pan 时按时间范围 lazy fetch，类似无限滚动列表
    Viewport-based / range-request fetching as user pans

🔗 跨图同步 / Cross-chart sync

  • 🟢 共享 hover state — Context / Zustand / Jotai 存当前 hover 的时间戳，所有 chart 订阅并画 crosshair
    Shared hover state in store; each chart subscribes and renders its own crosshair
  • 🟢 事件总线 — 一个 chart hover 时 dispatch event，其他 chart 监听（更松耦合）
    Event bus — looser coupling than shared store
  • 🟡 chart lib 自带 — Highcharts 有 Highcharts.Pointer.onContainerMouseMove + syncCharts 范例；Recharts / Chart.js 通常要自己接线
    Some libs have built-in sync (Highcharts demo); most need wiring
  • 性能要点：hover state 更新要 throttle 到 ~30 FPS，否则 mousemove 一秒触发 ~60 次 → 多 chart 都重渲

📱 Mobile 手势 / Mobile gestures (RN 强相关)

  • Web：用 Pointer Events（统一 mouse/touch/pen），或 PointerEvent + Pointer Lock；pinch 通过 2 个 active pointer 算距离差
  • React Native：react-native-gesture-handler + react-native-reanimated 的 worklet — 手势处理跑在 UI thread 上，不经过 JS bridge，60 FPS 流畅
    RN: gesture-handler + Reanimated worklets — gesture runs on UI thread, no JS bridge
  • Pinch zoom：scale 应用到 chart 的 x-axis domain，而不是 CSS transform（CSS transform 会糊）
    Pinch → x-axis domain change, NOT CSS transform (transform = blurry)
  • 双指 pan 和 单指 scroll 要区分（典型坑：单指 pan chart 会和页面 scroll 冲突，要 preventDefault / touch-action: none）

📋 两种 contract 形态对比 / Contract Shape Comparison

Offset / Limit (or page+pageSize)Cursor / Next-page token (keyset)
请求示例 / Request GET /transactions?offset=200&limit=20 GET /transactions?cursor=eyJ0c...&limit=20
响应 / Response { items: [...], total: 5234, page: 11 } { items: [...], nextCursor: "eyJ0c..." | null }
能否跳页 / Random page jump ✅ 可以直接跳到第 47 页 ❌ 只能 forward / backward 顺序翻
Total count ✅ 容易拿（一次 COUNT(*)） ⚠️ 不直接给，单独 endpoint 或近似值
新数据插入时的稳定性 🚩 不稳定：会跳记录或重复记录 ✅ 稳定：cursor 锚定具体行，不受插入影响

典型场景 admin 后台、需要跳页、数据相对静态 移动端 infinite scroll、feed、log、transaction list

🔬 SQL 层性能差异 / SQL-layer Performance（核心 / CORE）

① Offset / Limit 查询：

SELECT * FROM transactions
WHERE user_id = $1
ORDER BY created_at DESC, id DESC
LIMIT 20 OFFSET 10000;
  • DB 必须沿 index 顺序真的读出前 10,000 行然后丢掉，才能返回第 10,001~10,020 行
    DB must physically read 10,000 rows along the index then discard them before returning rows 10,001-10,020
  • 成本：O(offset + limit)——offset 越大越慢，线性恶化
  • 50 万 transaction 的用户翻到深页 → 几百毫秒甚至秒级
  • 错觉来源：很多人以为 "我加了 index 就 OK 了"——index 让 ORDER BY 不需要 sort，但不能让 DB 跳过 N 行；B-tree 不支持 "skip ahead"

② Cursor / Keyset 查询：

SELECT * FROM transactions
WHERE user_id = $1
  AND (created_at, id) < ($2, $3)   -- $2,$3 来自上一页最后一行
ORDER BY created_at DESC, id DESC
LIMIT 20;
  • 用 composite index (user_id, created_at DESC, id DESC) 直接index seek 定位到 cursor 位置，然后顺序读 20 行
    Index seek straight to the cursor position, then sequential read of 20 rows
  • 成本：O(log n + limit)——每页恒定，跟翻到多深无关
  • ⚠️ 关键：必须有 deterministic sort（sort key + tiebreaker，如 created_at + id）；只用 created_at 会在同一秒多条 transaction 时漏数据

💰 金融 transaction 列表为什么必须 cursor / Why Cursor for Transaction Lists

  • 银行 transaction 实时流入 → 用 offset 翻页时新插入会让所有 offset shift，导致跳记录 / 重复显示。
    Real-time inserts shift all offsets → users see skipped or duplicated rows on scroll.
  • 移动端只做 infinite scroll，不需要 jump page，cursor 没有任何缺点
  • Plaid 自家 /transactions/sync 用的也是 cursor (cursor 字段)——熟悉 Plaid 的人应该顺手提到

🎯 Cursor 设计细节 / Cursor Design Details（区分 "会做" vs "念词"）

  • Cursor 内容：sort key + tiebreaker（{ createdAt: "2024-...", id: "uuid" }），base64 encode 防止 client 篡改/解析
  • Sort 必须 deterministic：单字段 created_at 不够，必须 (created_at, id) 作 composite key
  • Bidirectional：要不要支持往回翻？API 设计选择，影响 cursor encoding（要不要带 direction）
  • Cursor 失效：如果 cursor 指向的行被删了怎么办？降级到 timestamp 邻近行 or 报错？
  • 不要让 client 解析 cursor——这是 opaque token，未来想换 schema 不破坏 client

🚫 反模式 / Anti-patterns

  • 🚩 给 transaction 列表用 offset 并且 "没遇到过问题" → 说明没真观察过用户在 prod 翻页
  • 🚩 cursor 只 encode 一个 timestamp，没有 tiebreaker → 同秒多条交易时会漏
  • 🚩 cursor 直接暴露 DB primary key，client 解析它做业务逻辑 → 等于把 internal schema 泄漏到 API
  • 🚩 "我加了 index 所以 offset 也快" → 没真跑过 EXPLAIN ANALYZE

📖 Q1 参考答案 / Reference

📦 Cursor 内容 / What goes inside

  • 🟢 Composite：sort key + tiebreaker——例如 { createdAt: "2024-03-15T10:23:45.123Z", id: "txn_uuid" }
    Composite: sort key + tiebreaker (e.g. createdAt + id)
  • 🟢 必须 deterministic：只用 createdAt 不够——同一毫秒可能多条交易（状态同步 / batch import），不加 tiebreaker 会漏数据或重复
    Must be deterministic — timestamp alone breaks on same-millisecond rows; tiebreaker is non-optional

🔐 Encoding / 编码选择

  • 🟢 Base64 (JSON 转码)——业界惯例。三个原因：
    • URL-safe (不需要 percent-encode JSON)
    • Opaque token——客户端看不懂就不会去解析、拼接、依赖它的格式（服务端未来能改 schema 不破坏 client）
    • 能加上 HMAC 签名防篡改（可选，金融场景推荐）
  • 🟡 加密一般不需要，cursor 不含敏感信息；需要防篡改用 HMAC 足够
  • 🚩 明文暴露 primary key（如 ?cursor=12345）——client 能猜下一个、能猜总量、能伪造请求，IDOR 风险

↔️ 方向 / Direction

  • Forward-only（infinite scroll 场景）：只给 nextCursor，cursor 体不带方向 — 90% 场景够用
    Forward-only for infinite scroll — simplest, covers 90% of cases
  • Bidirectional（需要"上一页"）：返回 nextCursor + prevCursor，cursor 里 encode direction 字段，服务端根据 direction 变 < 或 > 查询
  • 🟢 能主动说 "我当时只需要 forward，所以没加 bidirectional" = 明确 trade-off 意识

💥 Cursor 失效 / Invalidation

  • cursor 指向的行被删了怎么办？
    • 🟢 降级到邻近行：查询仍用 (created_at, id) < (cursor.ts, cursor.id)，本质上以 cursor 位置为上界，删行不影响后面查询 — 大部分场景这样处理就行
    • 🟢 或 sort key 选 immutable 字段（如 created_at 不变）——甚至不需要被指行存在
    • 🟡 返回 410 Gone / 400 + 让 client refresh — 复杂且 UX 差
  • cursor 指向的行被修改了怎么办（如 transaction 被 Plaid amend）？
    • 如果 sort key 变了（如按 amount 排序，amount 被改）→ 能跳记录。这也是为什么 sort key 要选 immutable 字段

🚫 反模式 / Anti-patterns

  • 🚩 cursor = 明文 primary key → client 可以猜 / 伪造 / 玩 IDOR
  • 🚩 cursor 是明文 JSON → client 会去解析它、依赖 schema，你后面不能改
  • 🚩 单字段 timestamp 没 tiebreaker → 同秒多条 transaction 会漏
  • 🚩 拿 updated_at 这种会变的字段做 sort key → 行被更新后 cursor 跳项

2️⃣ 有没有想过 怎么 detect（client 自觉 reset 还是 server 端 fingerprint 校验？两者优劣？）。
3️⃣ Bonus：有没有意识到 每加一种 sort 维度 = 一个 composite index 维护成本——所以 prod 系统的 sort 选项通常是固定小集合，不是无限。
🚨 钓鱼信号：如果他直接说 "切了 sort cursor 还能用" 或 "我们就 disable 切 sort 避免问题"——这条简历 "established pagination contracts" 基本是吹的。
Interviewer note — There's no valid "old cursor still works" answer here. Listen for: (1) immediate "cursor must be discarded" + the reason (cursor anchors a position in the OLD ordering, that position has no meaning in the new one); (2) detection strategy (client-side reset vs server-side fingerprint, trade-offs); (3) bonus: awareness that each sort dimension = a composite index to maintain, so prod systems offer a small fixed set. 🚨 Tell signals: "cursor still works" or "we disabled sort switching" → the "established pagination contracts" claim is likely bluff.
📖 Q2 参考答案 / Reference

🎯 核心结论 / Bottom line

Sort 或 filter 一变 → cursor 必须作废，从第一页重新开始。原因：cursor 本质是 "在按 X 排序、过滤条件 Y 下的 result set 中，我读到了哪一行"——切换 sort/filter 后这个 result set 不存在了，cursor 指向的位置无意义。
Sort or filter change → cursor must be invalidated, restart from page 1. A cursor is "where I am in the result set under sort=X, filter=Y" — changing either dissolves that set entirely.

🛠️ 三种实现选择 / Three implementation choices

方案 / Approach怎么做 / How优点 / Pros代价 / Cons
1. Client-side discard UI 切 sort/filter 时主动 reset，发不带 cursor 的请求 最简单，server 无需感知；contract 干净 依赖 client 自觉；老 client / 第三方 client 可能不守规矩 → 翻出乱序数据 server 完全不知道
2. Server-side fingerprint detection cursor 里 encode 当前 sort + filter 的 hash；request 时 sort/filter 跟 cursor 内 hash 不一致 → 返回 400 或 graceful 重置 + warning header 防御性强；老 client 也无法翻错；向后兼容容易演化 cursor schema 更复杂；要定义清晰的 error semantics（4xx vs 自动 reset）
3. Cursor namespace per sort 每种 sort 用独立 cursor 类型（如 cursor_by_date vs cursor_by_amount 是不同 schema） 类型系统帮忙强制隔离；编译期就 catch 误用 API surface 复杂；client 要同时维护多个 cursor 状态

生产系统最常见组合：方案 1 + 方案 2——client 主动 reset 是 happy path，server-side fingerprint 是兜底防御。
Production sweet spot: 1 + 2 — client reset is the happy path, server fingerprint is the defensive net.

🗂️ Index 层的连带成本 / Index-side cost

  • 每加一种 "可排序维度" 理论上需要一个 composite index 才能保证 cursor 走 index seek：
    • (user_id, created_at DESC, id DESC) — 按日期排
    • (user_id, amount DESC, id DESC) — 按金额排
    • (user_id, category, created_at DESC, id DESC) — 按类别 + 子排序
  • "任意 filter 组合 × 任意 sort 都建 index" = 不可能 → 写性能崩溃 + 存储爆炸 + planner cardinality 估算崩
  • 🟢 真做过的人会主动说："我们只暴露 2-3 种 sort 选项，加 sort 维度的代价是要新建 + 维护 index，不是免费的"
    "We expose only 2-3 sort options because each new sort means a new composite index to build & maintain — it's not free"
  • 🟡 折中：常用 sort 建 index，长尾 sort 走 seq scan / partial index / 接受慢，并 PM 沟通 UX 取舍

⏰ Filter snapshot 问题 / Filter snapshot pitfall（隐藏的金融坑）

  • 用户选 filter "Last 30 days"——是 server-side 每次解析成 NOW() - INTERVAL '30 days'，还是 client 把它解析成 absolute timestamp 传过来？
  • 🚩 如果 server 每次重新解析 NOW()，翻到下一页时 NOW() 已经变了 → 窗口边界滑动 → cursor 指向的 "30 天前" 那条 row 可能已经飘出窗口 → 跳行 / 重复 / 空白页
  • 🟢 正确做法：first page request 时 server 把 "30 days" 解析成一对 absolute timestamps，encode 进 cursor 一起带，后续翻页都用这同一组 timestamps → 窗口 "冻结"
  • 这跟 Plaid /transactions/get 的 start_date/end_date 必须是 absolute 同源——是分布式 / 翻页系统的通用要求
    Same reason Plaid /transactions/get requires absolute start/end dates, not relative — paginated time windows must be frozen on first request

🔧 Cursor schema 演化 / Backward-compat hook（呼应简历）

  • 未来新增 sort/filter 维度 → cursor schema 必然演化
  • 建议 cursor 里 encode version 字段：{ v: 2, sort: 'amount_desc', filter_hash: 'abc...', pos: {...} }
  • 解码失败 / version 不识别 → graceful degradation：返回 first page + warning header（如 X-Cursor-Reset: schema-evolved）
  • 简历明确说 "backward-compatible REST APIs" → 这是 backward-compat 在 cursor 这一层的具体含义——不是 endpoint URL 不变，而是老 client 拿老 cursor 也不会崩
    Resume says "backward-compatible REST APIs" — at the cursor layer that means "old clients with old cursors never crash", not just "endpoint URLs don't change"

🚫 反模式 / Anti-patterns

  • 🚩 切 sort 后仍把老 cursor 发过去 server 静默接受 → 结果集错乱（跳行 / 重复 / 空白页）；client 完全不知道
  • 🚩 filter "Last 30 days" 解析成 server-side NOW() 不放进 cursor → 翻页时窗口滑动 → 边界飘 → 跳记录
  • 🚩 每加一种 filter/sort 维度就建一个新 index，从不下线老 index → 写性能逐步崩溃 + planner cardinality 估算崩
  • 🚩 "我们前端直接禁掉 sort 切换" → 回避问题不是解决；UX 极烂；声称 "established pagination contracts" 但实际只有一种 ordering = 没设计 contract，只是写了一个 endpoint
  • 🚩 cursor 不 encode sort/filter fingerprint，依靠 client 自觉 — 第三方 client / 老版 app / postman 调试者必然踩坑

🎯 Q1 · ⚡ Pull-to-refresh + In-flight 翻页 / Pull-to-refresh vs In-flight Pagination ~3 min

🎤 提问 / Ask · Q1
"Real-world scenario. User is scrolling and your app is mid-flight fetching page 5. At the same time they yank down and pull-to-refresh. What actually happens in your app? How did you handle that race condition?"
"一个真实场景。用户正在 scroll，你的 app 正在 in-flight fetch 第 5 页。同时他下拉 pull-to-refresh。你的 app 里面实际会发生什么？这个 race condition 你怎么处理的？"
⚠️ 面试官小提示 — 这题金在"实际会发生什么"。真上过线的人会先描述 bug（"如果什么不做，list 会变成 [refreshed top] + [stale page 5]，跳记录或重复"），再说解决方案。直接开始背方案不提 bug 本身 = 🚩 没观察过。
Interviewer note — The gold is in "what actually happens". People who've shipped this describe the bug first ("if you do nothing, list becomes [refreshed top] + [stale page 5] → skip or dupe"), then the fix. Jumping straight to solutions without naming the bug = 🚩 never observed it.
📖 Q1 参考答案 / Reference

🐛 不处理的 bug / The naive bug

  • 时间线：T0 fetch page 5 发出 → T1 用户 pull-to-refresh → T2 refresh response 到，list 变成最新顶部 → T3 page 5 response 到，被 append 在末尾
  • 结果：list = [refresh 后的顶部（含新交易）] + [stale page 5（基于旧状态的 cursor / offset）] → 跳记录 / 重复记录 / 空洞
  • 重点：这不是 offset vs cursor 的问题——cursor 同样会中招，因为如果 refresh 后 cursor space 变了，老 cursor 的 page 5 结果也 stale

✅ 解决方案 / Solutions（从简到繁）

  • 🟢 方案 1 — In-flight cancellation (AbortController)：refresh 触发时 abort 所有 in-flight pagination requests，然后从第一页重拉。最干净。
    Cancel in-flight pagination via AbortController on refresh — cleanest
  • 🟢 方案 2 — Request generation counter：每次 refresh 让 generation++，response 回来检查是否 generation === current，不是就丢。适合不能 cancel 的场景。
    Bump a generation counter on refresh; discard responses whose generation ≠ current
  • 🟢 方案 3 — React Query / SWR 的 queryKey 机制：refresh 让 queryKey 变（或 invalidate），旧 query 自动 stale，response 不会被接受。如果她用了这些库，问她是否意识到这个机制。
    Or rely on React Query / SWR queryKey invalidation — ask if she actually understands the mechanism
  • 🟢 方案 4 — Tag request with cursor state：in-flight request 打上发起时的 cursor，response 到时比对 current cursor state，不匹配则丢。最符合 cursor pagination 设计哲学

📱 Mobile-specific 坑 / Mobile pitfalls

  • RN 原生 fetch 不像 React Query 那样自动丢 stale response — 容易踩
  • 慢网络下 page 5 和 refresh 同时 in-flight 是常态，不是 edge case
  • 用户快速 scroll 可能同时触发 page 4 / page 5 / page 6 三个 in-flight requests — 顺序返回不保证
  • 冷启动 + auto-refresh-on-focus 同时发生也会踩

🚫 反模式 / Anti-patterns

  • 🚩 "我没遇到过 race condition" → 从未观察过慢网用户
  • 🚩 "React Query 自己处理的" 但说不出 queryKey / invalidate 机制 → 只是使用者，不是设计者
  • 🚩 "我在 loading 时 disable pull-to-refresh" → 回避问题不是解决，UX 也差（用户需要能随时拉新）
  • 🚩 blindly append 所有 response 不检查是哪个 request 发的 → 必踩 bug
