📋 题面 / Problem Statement

English

Title: Implement Matrix Multiplication

Problem:

Given two matrices A and B of real numbers (double precision), implement a function multiply(A, B) that computes and returns their matrix product C = A × B.

  • A has dimensions m × k (m rows, k columns).
  • B has dimensions k × n.
  • The product C has dimensions m × n, where:
C[i][j] = Σp=0..k-1 ( A[i][p] · B[p][j] )

Input specification

  • A: an m × k matrix of doubles, with m ≥ 0 and k ≥ 0.
  • B: a k × n matrix of doubles, with n ≥ 0.

Output specification

  • C: an m × n matrix of doubles.
  • If any input matrix is empty (zero rows or zero columns), return an empty matrix.

Requirements

  1. The function MUST handle non-square matrices (do not assume m == k == n).
  2. The function MUST verify that A's column count equals B's row count. If they do not match, signal an error (throw std::invalid_argument in C++ / raise ValueError in Python).
  3. Implement the algorithm directly. Do NOT delegate the multiplication to a numerical library (e.g., numpy.matmul, Eigen, BLAS, cuBLAS). Library calls are out of scope for this exercise.
  4. Focus on correctness. Performance optimization is addressed in a subsequent phase.

  • B：维度为 k × n 的 double 矩阵，n ≥ 0。

输出规范

  • 若任一输入矩阵为空（零行或零列），返回空矩阵。

📖 C++ 参考答案 / C++ Reference Answer C++17 
#include <vector>
#include <stdexcept>

using Matrix = std::vector<std::vector<double>>;

Matrix multiply(const Matrix& A, const Matrix& B) {
    // ── Step 1: 空输入短路 ──────────────────────────────
    if (A.empty() || A[0].empty() || B.empty()) {
        return {};
    }

    // ── Step 2: 提取维度 ────────────────────────────────
    const size_t m = A.size();         // A 的行数
    const size_t k = A[0].size();      // A 的列数 = B 的行数
    const size_t n = B[0].size();      // B 的列数

    // ── Step 3: 维度校验 ────────────────────────────────
    if (B.size() != k) {
        throw std::invalid_argument(
            "dim mismatch: A is " + std::to_string(m) + "x" + std::to_string(k) +
            ", B is " + std::to_string(B.size()) + "x" + std::to_string(n));
    }

    // ── Step 4: 分配输出矩阵，全部初始化为 0.0 ──────────
    Matrix C(m, std::vector<double>(n, 0.0));

    // ── Step 5: 三重循环计算 ───────────────────────────
    for (size_t i = 0; i < m; ++i) {           // 遍历 C 的每一行
        for (size_t j = 0; j < n; ++j) {       // 遍历 C 的每一列
            double acc = 0.0;                  // 局部累加器
            for (size_t p = 0; p < k; ++p) {   // 点积：A 第 i 行 · B 第 j 列
                acc += A[i][p] * B[p][j];
            }
            C[i][j] = acc;                     // 写回 C[i][j]
        }
    }

    return C;
}
📝 全流程解说 / Step-by-step walkthrough

Step 1 — 空输入短路

三个条件：

  • A.empty() → A 一行都没有（m=0）
  • A[0].empty() → A 有行但列数为 0（k=0）
  • B.empty() → B 一行都没有（k=0）

任何一个成立都直接返回 {}（构造空 Matrix），避免后续访问 A[0].size() 时 segfault。

Any empty input short-circuits to {}. Without this guard, accessing A[0].size() below would crash.

Step 2 — 提取维度

用 size_t（无符号整型）保存维度，匹配 STL 容器 .size() 的返回类型，避免编译期 sign-compare warning。const 修饰强调这三个值不应被修改。

Use size_t to match vector::size() return type — avoids sign-compare warnings. const documents intent.

Step 3 — 维度校验

要求 B.size() == k，即 B 的行数等于 A 的列数。不满足就抛 std::invalid_argument，并把具体维度写进错误信息便于调试。

Validate B.size() == k. Throw std::invalid_argument with concrete dimensions in the message for debuggability.

Step 4 — 输出矩阵分配

Matrix C(m, std::vector<double>(n, 0.0)) 这个构造器调用：

  • 外层 vector 大小为 m
  • 每个元素是一个内层 vector<double>，大小 n，所有元素初始化为 0.0
注意 / Note: 内层 vector 是独立堆分配的，行与行之间内存不连续——这是嵌套 vector 数据结构的固有特性，会在 P3 blocking 讨论中重新出现。
Inner vectors are independently heap-allocated; rows are NOT contiguous in memory. This becomes a perf concern in P3 (blocking).

Step 5 — 三重循环

经典 i-j-p 顺序：

  • 外两层 i, j 锁定要算 C 的哪一格
  • 最内层 p 是点积的求和变量
  • 关键工程细节：用局部变量 acc 累加，最后一次性写入 C[i][j]。比直接 C[i][j] += ... 减少对 C[i][j] 的反复 indirection 访问（虽然现代编译器开 -O2 后通常能识别并优化掉，但显式写出是 senior 习惯）。

Classic i-j-p triple loop. Key engineering detail: a local acc accumulator avoids repeated indirection through C[i][j]. Modern compilers often hoist this with -O2, but writing it explicitly is a senior habit.

🐍 Python 参考答案 / Python Reference Answer Python 3.10+ 
from typing import List

Matrix = List[List[float]]

def multiply(A: Matrix, B: Matrix) -> Matrix:
    # ── Step 1: 空输入短路 ──────────────────────────────
    if not A or not A[0] or not B:
        return []

    # ── Step 2: 提取维度 ────────────────────────────────
    m = len(A)              # A 的行数
    k = len(A[0])           # A 的列数 = B 的行数
    n = len(B[0])           # B 的列数

    # ── Step 3: 维度校验 ────────────────────────────────
    if len(B) != k:
        raise ValueError(
            f"dim mismatch: A is {m}x{k}, B is {len(B)}x{n}")

    # ── Step 4: 分配输出矩阵，全部初始化为 0.0 ──────────
    C = [[0.0] * n for _ in range(m)]

    # ── Step 5: 三重循环计算 ───────────────────────────
    for i in range(m):                          # 遍历 C 的每一行
        for j in range(n):                      # 遍历 C 的每一列
            acc = 0.0                           # 局部累加器
            for p in range(k):                  # 点积
                acc += A[i][p] * B[p][j]
            C[i][j] = acc                       # 写回

    return C
📝 全流程解说 / Step-by-step walkthrough

Step 1 — 空输入短路

Python 的 truthiness：空 list [] 为 False。not A 即 len(A) == 0；not A[0] 即 A 的第一行是空 list（k=0）。判断顺序不能颠倒（先确认 A 非空才能访问 A[0]）。

Python truthiness: empty list is falsy. Order matters — must check not A first before accessing A[0].

Step 2 — 提取维度

直接 len()。Python 没有 unsigned int，统一用普通 int。

Step 3 — 维度校验

用 f-string 内插具体维度。抛 ValueError（Python 的内置数值错误异常）。

Step 4 — 输出矩阵分配

[[0.0] * n for _ in range(m)] 是列表推导式：

  • 外层生成 m 个独立列表
  • 每个内层 [0.0] * n 生成长度 n 的全零 list
⚠️ 新手常见陷阱 / Common pitfall: 写成 [[0.0] * n] * m 是错的——这样 m 个内层 list 是同一个对象的引用，改一个全变。必须用列表推导式让每一行是独立的 list。这本身就是一个面试观察点。
Writing [[0.0] * n] * m creates m aliased references to the SAME inner list. Mutating one mutates all. Always use the list comprehension form. This pitfall is itself an observation point.

Step 5 — 三重循环

与 C++ 完全对应。range(k) 生成 0 到 k-1。Python 没有 ++，自然用 for p in range(k)。

追问 1：存储 layout & 访存代价 / Storage Layout & Memory Access Cost 2-3 min ⭐⭐ Medium

🎯 锚点 / Anchor

提问 / Question

EN: "Your Matrix is vector<vector<double>>. Two questions:
(a) Is this row-major or column-major in memory, and how do you know?
(b) Now look at your innermost line acc += A[i][p] * B[p][j]. Given that layout and your i-j-p loop order, between these two reads — A[i][p] and B[p][j] — which one is the bottleneck on a 1024×1024 matrix, and why?"
中: "你的 Matrix 是 vector<vector<double>>。两个问题：
(a) 这种存储是 row-major 还是 column-major？你怎么判断？
(b) 看你的最内层 acc += A[i][p] * B[p][j]。基于这个 layout 和你的 i-j-p 循环顺序，这两次读取——A[i][p] 和 B[p][j]——在 1024×1024 输入下哪个是瓶颈？为什么？"
📖 标准答案 / Expected Answer

(a) Layout 部分 / Layout part：

中: Row-major——但有一个重要 caveat：vector<vector<double>> 的每一行内部是连续的（一个 vector<double> 就是堆上一段连续 buffer），但行与行之间不一定连续——外层 vector 持有的是 m 个独立的 vector<double> 对象，每个对象的 data 指针分别走 new[]，它们在堆上的位置是分散的。

判断依据：A[i][p] 这个语法 = 先 A[i] 取出第 i 个 vector<double>，再 [p] 取该行第 p 个元素 → 先按行索引，所以是行优先。

真正的 row-major flat layout 是 vector<double>(m*n) + [i*n+j] 索引——那才是连行间也连续。vector<vector<double>> 只能说"按行存放、行内连续"。

EN: Row-major — but with an important caveat: each row in vector<vector<double>> is contiguous (one vector<double> = one contiguous heap buffer), but rows are not contiguous with each other. The outer vector holds m independent vector<double> objects; each one's data pointer goes through its own new[], scattered on the heap. True row-major flat layout would require vector<double>(m*n) with [i*n+j] indexing — that's the only way to get inter-row contiguity. Evidence: A[i][p] = first index by row i, then by column p → row-first.

(b) 访存代价部分 / Access cost part：

中: 最内层是 p 递增：

  • A[i][p]：i 固定、p 递增 → 在同一行内连续访问 → stride-1，cache 友好。一个 cache line 64 字节 = 8 个 double，每 8 次访存才一次 cache miss。
  • B[p][j]：j 固定、p 递增 → 跨不同行访问，而行间又不连续（见 (a) 的 caveat）→ 每次访存几乎都 miss，n 大时还伴随 TLB miss。

结论：B 是瓶颈。在 n=1024（每行 8KB）的量级，B 的访问会比 A 慢 8-50 倍（取决于 L1/L2/L3 命中状况）。

EN: Inner loop increments p. A[i][p]: i fixed, p increments → walks along the same row → stride-1, cache-friendly (a 64-byte cache line holds 8 doubles, so 1 miss per 8 accesses). B[p][j]: j fixed, p increments → walks across rows, and rows aren't contiguous (see caveat in (a)) → nearly every access misses, plus TLB misses at large n. Conclusion: B is the bottleneck. At n=1024 (8KB per row), B reads are 8-50× slower than A reads depending on cache hierarchy hit rates.

📖 标准答案 / Expected Answer

代码（C++）/ Code (C++)

Matrix multiply(const Matrix& A, const Matrix& B) {
    const size_t m = A.size();
    const size_t k = A.empty() ? 0 : A[0].size();
    const size_t n = B.empty() ? 0 : B[0].size();
    if (!B.empty() && B.size() != k)
        throw std::invalid_argument("dim mismatch");

    Matrix C(m, std::vector<double>(n, 0.0));   // 必须预先置 0
    for (size_t i = 0; i < m; ++i) {
        for (size_t p = 0; p < k; ++p) {
            const double a = A[i][p];             // hoist 出内层
            for (size_t j = 0; j < n; ++j) {
                C[i][j] += a * B[p][j];           // B[p][·] 、 C[i][·] 都是 stride-1
            }
        }
    }
    return C;
}

与原版的三个区别 / Three differences from original:

  1. 循环顺序：i-j-p → i-p-j（交换后两层）
  2. 累加方式：原版用 acc 累加完后赋值给 C[i][j]；新版用 C[i][j] +=——所以 C 必须预先置 0（原代码的 vector(n, 0.0) 刚好满足）
  3. 提取不变量：const double a = A[i][p]; 提到最内层外面（编译器一般会做，主动写出体现对热点的控制意识）

为什么这能解决：三层解释 / Why this works — three layers

第一层 · 访问 pattern 对比表 / Layer 1 · Access pattern comparison

访问原版 i-j-p（内层是 p）F2 版 i-p-j（内层是 j）
A[i][p]i 定、p 增 → stride-1 ✅i、p 都定 → scalar（提取到寄存器）✅✅
B[p][j]j 定、p 增 → stride-n ❌p 定、j 增 → stride-1 ✅
C[i][j]i、j 都定 → scalar（acc 在寄存器）✅✅i 定、j 增 → stride-1 写入 ✅

关键洞察：F2 用一次 "降级"（C 从 scalar 降到 stride-1）换了一次 "升级"（B 从灾难性的 stride-n 升到 stride-1）。B 的修复是巨大的赢，C 的降级是轻微的输。净结果：三个访问全部是 stride-1 或 scalar，没有一个是 stride-n。
Key insight: F2 trades one "downgrade" (C: scalar → stride-1) for one "upgrade" (B: stride-n → stride-1). B's fix is a huge win, C's downgrade is a tiny loss. Net result: all three accesses are stride-1 or scalar; none is stride-n.

第二层 · 硬件为什么 stride-1 这么快 / Layer 2 · Hardware mechanisms (why stride-1 wins)

  1. Cache line 利用率（最重要）：CPU 读 B[p][j] 时硬件强制载入一条 64 字节 cache line = 8 个 double。原版下一次跳到 B[p+1][j]，距离 n×8 字节，刚载入的另外 7 个 double 一个都没用上就被淘汰——87.5% 带宽浪费。F2 版载入后紧接着 7 次迭代刚好用掉 7 个值 → 100% 利用率 → 8× 内存带宽提升。
  2. 硬件 prefetcher（隐藏延迟）：stride-n（n=1024 时 = 8KB 跳跃）超出多数 CPU prefetcher 预测能力，prefetcher 直接放弃，每次访问都要等 ~100ns DRAM 延迟。stride-1 是 prefetcher 的最理想场景，能完全隐藏延迟。
  3. TLB 压力：stride-n 下每个 B[p][j] 在不同 page，很快手爆 L1 dTLB（通常 64 项）→ page walk 又加几十 ns。stride-1 TLB 命中率几乎 100%。
  4. Load 流水线：stride-n 下 load 之间锘锘 cache miss 依赖，LSU 被串行化；stride-1 可以满发多发 load。

第三层 · 数学上为什么合法 / Layer 3 · Why this is a valid transformation

原版和 F2 版计算量完全一样（都是 m×k×n 次乘加），只是顺序变了。这本质是矩阵乘法的两种数学公式：

  • 原版 = 点积公式：Cij = Σp Aip·Bpj，一次算出 C 的一个元素（外层 固定 (i,j)，内层 走 p）
  • F2 版 = 外积公式：C = Σp A:,p ⊗ Bp,:，每个 p 贡献一个秩 1 矩阵，累加到 C（外层 走 p，内层 一次更新 C 的所有元素）

这就是为什么 F2 需要 C 预先置 0、原版不需要。BLAS 库全有并不是一个公式一路走到黑——他们在不同维度对 GEMM 拆分重组，就是为了让计算顺序贴合硬件缓存层次。

EN: Original and F2 do exactly the same arithmetic (m×k×n multiply-adds) — only the execution order changes. This is fundamentally two mathematical formulations of matmul: dot-product (one C element at a time) vs outer-product (rank-1 update of all C elements per p). That's why F2 requires C pre-zeroed and original doesn't. BLAS libraries exploit exactly this freedom — they recast GEMM in different layouts to match the hardware memory hierarchy.

📖 标准答案 / Expected Answer

(1) 理想 layout / Ideal layout

一个 单一、连续、按行优先（row-major）展开的 buffer，再加 (rows, cols) 两个元数据描述形状（生产级再加 leading dimension ld 描述行步长，以支持 sub-matrix slice）。逻辑上是 2D，物理上是 1D 数组——(i, j) 元素位于偏移 i * cols + j（或 i * ld + j）。
A single, contiguous, row-major flattened buffer, plus (rows, cols) metadata to describe shape (production-grade adds leading dimension ld for sub-matrix slicing). Logically 2D, physically a 1D array — element (i, j) sits at offset i * cols + j (or i * ld + j).

反例：vector<vector<double>> / Python 的 list-of-list——每一行是一次独立分配，行与行之间内存地址不连续，外层是一个指针数组指向各行。
Counter-example: vector<vector<double>> / Python list-of-list — each row is a separate allocation, rows are not contiguous in memory, and the outer layer is just a pointer array pointing to each row.

(2) 为什么这种 layout 让乘法变快 / Why this layout makes multiply fast

期望候选人讲具体机制，不止 "cache-friendly"。覆盖 ≥3 条算合格 / Expect specific mechanisms, not just "cache-friendly". ≥3 of the following is a pass：

  1. 一次性堆分配 / Single heap allocation：m * n 个 double 走一次 malloc；vs nested 版要 m + 1 次。减少 allocator 开销 + 避免堆碎片。
  2. 跨行内存连续 / Inter-row contiguity：硬件 prefetcher 能从第 i 行尾部预取到第 i+1 行头部；TLB 命中率高、可享用 huge pages。
  3. 指针 indirection 消失 / No pointer indirection：A(i,j) = 1 次乘加 + 1 次 load；nested 版 = 1 次 load 行指针 + 1 次 load 数据，少走 1 跳内存。
  4. SIMD load 友好 / SIMD-friendly load：AVX2 一次连续读 4 个 double、AVX-512 读 8 个、NEON 读 2 个；连续 buffer 才能用 aligned load 指令，nested 版跨行边界会断。
  5. 可控的 alignment / padding / Controllable alignment：可以让每行起始地址对齐到 cache line（64B）或 SIMD register 宽度，避免 false sharing 和未对齐 load 惩罚；nested 版受 allocator 摆布。
  6. 直接喂 BLAS / cuBLAS / GPU / Directly feedable to BLAS：这些库的 API 全部是 (double* ptr, int ld) 形式；连续 buffer 可以零拷贝 cudaMemcpy 到 device，nested 版必须先拍平。
  7. 跨语言零拷贝 / Zero-copy cross-language：一个连续 buffer 可以通过 Python buffer protocol、Rust slice、CUDA pointer、网络协议不拷贝地传递。

📖 标准答案 · F4a 诊断 / Expected Answer — F4a Diagnosis

必须先算数量级 / Must establish magnitudes first

  • 数据量：10000² × 8B = 800 MB per matrix；A+B+C 三个 = 2.4 GB
  • 计算量：10000³ = 10¹² FMA ≈ 1 TFLOP
  • 朴素单线程 CPU：~10 GFLOPS → 100 秒；优化后 ~80 GFLOPS（5GHz × AVX-512 × FMA）→ ~12 秒
  • 单 GPU 理论上限（V100 FP64 ~7 TFLOPS）→ ~0.14 秒

瓶颈结论 / Bottleneck conclusion

  1. Cache 进不下：800 MB 远超 L3（典型 8-64 MB）→ 每个元素都要从 DRAM 取
  2. B 矩阵被遍历 N 次（接 F1）→ 内存带宽是 wall，memory-bound
  3. 加分点：NUMA / page fault / TLB miss——大分配可能跨 NUMA node、首次访问触发 minor fault；TLB 覆盖不住 800 MB（典型覆盖 ~2 MB）

判断："memory-bound, 单线程不可接受"——senior 必须秒答的诊断。

📖 标准答案 · F4c GPU 深挖 / Expected Answer — F4c GPU Deep Dive

核心思路（必答）/ Core idea (must answer)

GPU = 算力批发店——进货成本（PCIe 传输）+ 零售单价（GPU 算力）。判断值不值得上 GPU，两个条件要同时满足：

  1. 算得多：计算量大到能"摊平"PCIe 传输成本
  2. 数据复用：搬一次能算多次（训练循环 / batch matmul）

关键数字：PCIe 4.0 ~12 GB/s 单向 → 800MB 单程 ~67ms；如果只算一次就丢，PCIe 时间可能 > 计算时间，反而比 CPU 慢。

什么场景值得上 / When GPU pays off

  • N > 10000 单算（数据搬运摊得平）
  • Batch matmul（一次搬多个矩阵）
  • 训练循环（同一份权重反复用 N 个 epoch）
  • matmul 本身是 GPU 天选工作——算术强度极高（O(N²) 数据 vs O(N³) 计算，每搬 1 byte 算 N 次），整个深度学习革命就是建立在这之上

什么场景不值得（关键考察）/ When GPU is NOT worth it (key probe)

  • 单次中等规模 + 结果立刻给 CPU 用：PCIe 往返时间 > GPU 计算时间
  • 大量小矩阵（N<100）：每个 kernel launch 的 ~5μs 开销盖过算力优势——这种用 batched GEMM 或干脆 CPU
  • 没有 GPU 硬件 / 部署在 edge / 客户没买 GPU：工程现实
  • 开发成本不划算：一次性脚本，用 BLAS 几行搞定，没必要折腾 GPU 环境

如果真的上 GPU，怎么用 / How to actually use GPU

  1. 第一选择：cuBLAS（cublasDgemm）——几乎永远更快，几十人年调优
  2. 第二选择：高层框架——PyTorch / JAX / CuPy，自动选 backend
  3. 自写 kernel 仅在：fused op（matmul + activation）、非标准 shape、研究新算法

自写 naive kernel 思路（候选人主动写代码时）/ Naive kernel sketch (if candidate volunteers)

  • Grid / Block：2D grid，每个 thread 算一个 C[i][j]；典型 16×16 thread block
  • Naive kernel 瓶颈：每个 thread 要从 global memory 读 2N 个元素，memory-bandwidth bound（和 CPU 朴素版同样的病）
  • 救命招：shared memory tiling——每个 block 协作把一对 tile 加载到 shared memory，所有 thread 复用 → 减少 N/tile 倍的 global memory 流量
  • 进阶：tensor cores（FP16/TF32 mixed precision，A100 上 ~300 TFLOPS）、多级 tiling（thread tile → warp tile → block tile）、CUTLASS
