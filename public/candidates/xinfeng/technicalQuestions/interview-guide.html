➜ 追问 1 / Follow-up 1 · 🎯 GPU 显存 & Voice 生命周期 / GPU Memory & Voice Lifecycle ~3 min

📌 关联背景 / Background: 单张 H100 = 80GB 显存；假设每个 voice 模型 ~1GB，单卡最多装 ~70 个 voice；500+ voice 必须动态调度。
A single H100 has 80GB of memory. If each voice ~1GB, one card fits ~70 voices max. 500+ voices must be dynamically scheduled.

🎤 提问 / Ask
"500 voice models clearly can't all fit in a single GPU's memory at the same time. How did you manage their lifecycle — what's hot, what's cold, when do you load, when do you evict?"
"500 个 voice 模型加起来肯定塞不进任何一张 GPU 的显存。你是怎么管理它们的生命周期的——什么算热、什么算冷、什么时候 load、什么时候 evict？"
🚨 追问 1 卡壳引导 / Follow-up 1 Hints When Stuck

Light

"Voice 的请求分布在实际中是什么样的？是均匀的还是长尾的？"

"如果 Aria 这种热门 voice 占了 30% 流量，而另一个 voice 一天只有 1 个请求——它们应该被一样对待吗？"

Heavy

"Think about it as a cache problem — you have limited GPU memory (the cache), and need a policy to decide what stays and what gets kicked out."
"可以把它当作一个缓存问题——GPU 显存有限（就是 cache），你需要一个策略决定什么留下、什么被踢出。"
📖 追问 1 面试官参考答案 / Follow-up 1 Interviewer Reference Answer
⚠️ 面试官专用 — 不要念给候选人。用于评估候选人回答的完整度。
Interviewer-only — Do not read to candidate. Use to evaluate answer completeness.

✅ 热/冷分层 / Hot & Warm Pool Tiering

  • Hot pool：Top-N 热门 voice 常驻 GPU 显存，绝不 evict
    Top-N hot voices pinned in GPU memory, never evicted
  • Warm pool：按需 load，可被 evict
    On-demand load, evictable
  • 典型比例：~20% 热 voice 吃 80% 流量（Pareto 分布）
    ~20% hot voices serve 80% traffic (Pareto)

✅ 淘汰策略 / Eviction Policy

  • LRU：最近最少使用——简单但忽略权重
    Simple but ignores priority
  • Weighted LRU / LFU：考虑客户优先级、付费等级、SLA 等级
    Account for customer priority, paid tier, SLA
  • Watermark：显存达 80% 触发 evict；evict 到 60% 停止
    Trigger evict at 80% memory; stop at 60%

✅ Pod 调度策略 / Pod Scheduling

  • 典型架构：不是每个 voice 一组 pod（500 组太浪费）；而是一个 pod 同时承担多个 voice，按 GPU 显存容量打包
    NOT one pod per voice (too wasteful) — one pod hosts multiple voices, packed by GPU memory capacity
  • 新 voice 来时，router 决定丢给哪个有空闲的 pod，触发该 pod 的 model swap
    When new voice comes, router picks a pod with free capacity, triggers model swap on that pod

✅ Evict 时的并发 / Eviction Concurrency Safety

  • 正在跑请求的 voice 不能 evict → 引用计数 (refcount) 或 readers-writer lock
    Voice with in-flight requests can't be evicted → refcount or RW lock
  • Evict 前 drain 等待最后一个请求完成
    Drain in-flight requests before evicting

✅ 长尾 voice (Custom Voice) 特殊处理 / Long-tail Handling

  • 用 ZeroScaler 缩到 0，请求来时再起
    Scale to zero with ZeroScaler, spin up on demand
  • SLA 合同允许冷启动延迟（写在 Custom Voice 条款）
    SLA contract permits cold-start latency (written in Custom Voice terms)

➜ 追问 2 / Follow-up 2 · 🔀 请求路由 / Request Routing ~3 min

📌 关联背景 / Background: 客户请求里指定 voice ID。K8s 默认 round-robin LB 会让请求随机落到任何 pod 上，cache 命中率惨不忍睹。
Request specifies a voice ID. K8s default round-robin LB scatters requests across pods, killing cache hit rate.

🎤 提问 / Ask
"When a client request comes in with a specific voice ID, how does it actually get routed to the right pod? Default K8s Service round-robin obviously wouldn't work — what did you build instead?"
"客户请求里指定了 voice ID——这个请求进来后，怎么知道该路由到哪个 pod？K8s 默认的 round-robin Service 显然不行——你们是怎么实现的？"

📖 追问 2 面试官参考答案 / Follow-up 2 Interviewer Reference Answer

✅ 路由层架构 / Router Architecture

  • Voice-aware Router：独立微服务 或 ingress 扩展（不能依赖默认 K8s Service）
    Standalone microservice or ingress extension — can't rely on default K8s Service
  • 维护 voice → pod 索引（哪些 pod load 了哪些 voice），存 Redis / etcd / 内存 + 心跳同步
    Maintains voice → pod index in Redis/etcd/in-memory with heartbeat sync
  • Router 自己 HA：多副本 + 索引共享存储，单点 fail 不影响请求
    Router itself is HA: replicas + shared index store

✅ Consistent Hashing on Voice ID

  • 同 voice 的请求 → 同 pod，让该 pod 的 KV-cache / 模型 warm-up 状态被复用
    Same voice → same pod, reusing KV-cache / warm-up state on that pod
  • Pod 增删时 only ~K/N 的请求需要重路由（vs 简单 hash 的 K 个全部重路由）
    Only ~K/N requests rerouted on pod churn (vs K with naive hashing)

✅ Cold Voice 处理 / Cold Voice Handling

  • 没人 load 的 voice → 选一个容量空闲 pod 触发 load → 路由到它
    No pod has it → pick free-capacity pod → trigger load → route there
  • Load 期间的请求可以排队或返回 retry-after
    In-flight requests during load can queue or return retry-after

✅ Pod 增删时的 Rebalance / Pod Churn

  • 新 pod 上线 → 索引同步 → router 开始把部分流量分给它
    New pod up → index sync → router starts directing partial traffic
  • 老 pod 下线 → drain 等 in-flight 请求完成 → 索引移除
    Old pod down → drain in-flight → remove from index

✅ Istio 在这里能干什么 / What Istio Brings

  • Istio 本身不懂 voice 业务语义，但能提供：mTLS、tracing、circuit breaker、retry
    Istio doesn't understand voice semantics, but provides: mTLS, tracing, circuit breaker, retry
  • Voice-aware routing 在 Istio 之上的 application 层
    Voice-aware routing sits above Istio at the application layer

➜ 追问 3 / Follow-up 3 · ⚡ 冷启动 / Cold Start ~2 min

📌 关联背景 / Background: 加载一个 ~1GB 模型从磁盘到 GPU 显存大约 5–30s。冷门 voice 直接命中冷启动，用户体验灾难。
Loading a ~1GB model from disk to GPU memory takes ~5–30s. Long-tail voices hit cold-start directly — a UX disaster.

🎤 提问 / Ask
"A voice that hasn't been used for a year suddenly gets a request. What's the cold-start latency you typically see? And how do you mitigate the user experience hit?"
"一个一年没人用的 voice 突然来了请求——你们一般看到的冷启动延迟是多少？用户体验上你们怎么 mitigate？"
📖 追问 3 面试官参考答案 / Follow-up 3 Interviewer Reference Answer

✅ 冷启动延迟分解 / Cold-start Latency Breakdown

  • 磁盘读 model artifact：几百 ms 到几 s（取决于 blob storage vs 本地 SSD）
    Disk read model artifact: 100ms - several seconds
  • GPU memcopy：~1GB 模型 PCIe 拷贝 ~100ms
    GPU memcopy: ~100ms for 1GB over PCIe
  • 模型 init / CUDA kernel warmup：首次推理 kernel 编译 + cache 1-5s
    Model init / CUDA kernel warmup: 1-5s on first inference
  • 合计：5-30s 是正常区间
    Total: 5-30s is the normal range

✅ Mitigation 手段 / Mitigation Strategies

  • Pre-warm pool：根据历史调用预测哪些 voice 要起，提前 load
    Predict from history which voices will be requested, pre-load
  • N+1 buffer pod：永远多保留 1 个 idle pod 接 cold request
    Always keep 1 idle pod for cold requests
  • 客户端流式返回：返回 streaming response 或 status，让客户知道"正在 load"
    Return streaming response or status so client knows it's loading
  • Fallback 小模型：用小 footprint 模型做 first-token 响应，大模型 background load
    Smaller-footprint model for first-token response, larger model background-loads
  • SLA 降级：Custom Voice 这种极冷场景接受冷启动，明确写进合同
    For extreme cold-start scenarios like Custom Voice, accept cold start in SLA contract

✅ 关联到 ZeroScaler / Tie to ZeroScaler

  • ZeroScaler 可以把极冷 voice 缩到 0 pod 节省成本
    ZeroScaler can scale extreme-cold voices to 0 pods to save cost
  • Trade-off：第一个请求必然冷启动 → 必须接受或用 fallback
    Trade-off: first request guaranteed cold start → must accept or use fallback

➜ 追问 4 / Follow-up 4 · 🚀 发布管线 / Release Pipeline ~2 min

📌 关联背景 / Background: 500 个 voice × 多版本 × 持续训练新版本 = 发布频率极高。手动发布在这个规模下不可能。
500 voices × multiple versions × ongoing retraining = extremely high release cadence. Manual release impossible at this scale.

🎤 提问 / Ask
"How do you manage releases across 500 voices? When a new voice gets trained, or an existing voice gets a v2 — what does the rollout workflow look like?"
"500 个 voice 你们怎么管发布？一个新 voice 训出来，或者一个老 voice 升级到 v2——上线流程是什么样的？"
📖 追问 4 面试官参考答案 / Follow-up 4 Interviewer Reference Answer

✅ Model Registry / Artifact 管理

  • Model Registry（类似 MLflow / Azure ML）：每个 voice 每个版本 = 一个 artifact，带 metadata（训练数据集、超参、quality 评分）
    Model Registry: each voice × version = one artifact with metadata (training data, hyperparams, quality score)
  • 不可变 artifact：发布后不改，新改动 = 新版本
    Immutable artifacts: once published, never modified — changes = new version

✅ 自动 QA / Automated Quality Gates

  • 训完即跑回归测试集：几十种文本场景（短句、长句、数字、外语词、特殊符号）
    Regression test suite runs on training completion: dozens of scenarios
  • 自动 MOS 评分（Mean Opinion Score）或用预训练判别模型打分
    Automatic MOS scoring or pretrained discriminator
  • 低于阈值直接 fail，不进 registry
    Below threshold = fail, never enters registry

✅ Canary / 灰度发布 / Progressive Rollout

  • Per-voice 灰度：v2 先放 1% → 5% → 25% → 100%
    Per-voice canary: 1% → 5% → 25% → 100%
  • 每档观察 24h，监控：P99 延迟 / 错误率 / 业务 MOS 分 / 用户复购率
    Hold each stage 24h, monitor P99/error rate/MOS/return rate
  • 路由层支持 voice 维度的流量切分（同一 voice 不同版本可以并存）
    Router supports per-voice version split (same voice different versions can coexist)

✅ 自动回滚 / Automated Rollback

  • 任一指标超阈值 → 一键回滚（router 切回老版本）
    Any metric over threshold → one-click rollback (router switches back)
  • 老版本 artifact 必须保留至少 N 天，方便回滚
    Keep old artifacts at least N days for rollback

✅ 多 region 协调 / Multi-region Coordination

  • 新版本不是一次性推到所有 region
    New version is not pushed to all regions at once
  • 典型：dev → staging → 1 个小 region → 主要 region → 全球
    Typical: dev → staging → 1 small region → primary regions → worldwide
