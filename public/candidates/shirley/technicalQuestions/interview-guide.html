📖 面试官参考答案 / Interviewer Reference — Q1 Project Context

🏢 Mark43 是什么 / What Mark43 is

  • 警务 SaaS 平台（RMS — Records Management System），给警察局用
  • 覆盖警务全流程：报案 → 立案 → 调查 → 证据 → 起诉
  • 客户 = 警察局（agencies），简历的 "5+ police agencies" 就是指这些
  • A SaaS RMS (Records Management System) used by police agencies — covers the full workflow from incident report to prosecution. "5+ agencies" in her resume = client police departments.

📜 Warrants 是什么 / What warrants are

  • 法律文件，由法官签发，授权警方执行强制行为
  • 三种主流：Arrest warrant（逮捕令）/ Search warrant（搜查令）/ Bench warrant（拘传令）
  • 关键性质：有法律效力—— 严格签发流程、可追溯、可审计、可打印盖章
  • Legal documents issued by a judge, authorizing police action (arrest / search / bench). Legally binding — strict workflow, auditable, printable.

📝 Warrants creation workflow 实际在做什么 / What the workflow actually does

典型多步表单流程 / Typical multi-step form workflow:

  1. 选关联案件（case lookup）
  2. 选 warrant 类型（arrest / search / bench）
  3. 填表单：suspect 信息、涉嫌罪名（charges）、Probable Cause 陈述、签发法官、限制条件、附件
  4. 上级审核（supervisor approval workflow）—— 状态机：Draft → Pending Review → Approved/Rejected
  5. 提交法院（export PDF / e-file）
  6. 法官签发后回写状态（Issued / Active）
  7. 执行后回填（Served / Returned / Expired）

🎯 Intern 4 个月合理能 own 的范围 / What an intern can realistically own in 4 months

  • 🟢 很可能：实现 Create Warrant 这一个页面/feature（React 表单 + 校验 + 提交到 API + 后端写 MySQL/ES）
  • 🟢 可能：实现 warrant 的列表/搜索页（ES 查询 + 分页 + 过滤）
  • 🟢 可能：参与状态机的前端展示
  • 🟡 不太可能完全自己 own：整套 schema 设计、ES mapping 设计、跨警局多租户架构（通常 senior/staff 定）
  • 🚩 红旗：如果她说"整个系统都是我设计的" —— 与 intern 身份/时长不符，应深入追问

➜ 追问 1 / Follow-up 1 · 🎯 MySQL & Elasticsearch 分工 / Why Two Systems ~7 min CORE
🎤 提问 / Ask
"Your bullet says you used both MySQL and Elasticsearch for this. Can you walk me through — what does each one actually do in the warrants workflow? What kind of data sits where? And why couldn't you just use one of them — say, MySQL alone? What does Elasticsearch buy you that MySQL doesn't?"
"你 bullet 里说同时用了 MySQL 和 Elasticsearch。你能讲讲——在 warrants workflow 里，这两个分别在做什么？什么数据放在哪？为什么不只用一个？比如只用 MySQL 不行吗？Elasticsearch 给了你 MySQL 给不了的什么？"
⚠️ 面试官提示 — 这是核心架构题，主要看三件事：
(1) 角色分工是否清晰：MySQL = source of truth（强一致、事务、审计），ES = 搜索副本（全文搜索、模糊匹配、复杂过滤、聚合）。
(2) 为什么 MySQL 不够：能否说出 MySQL 全文搜索（FULLTEXT / LIKE '%...%'）在 warrants 这种"模糊匹配 + 多字段组合 + 相关性排序"场景下不够用。
(3) 为什么 ES 不够：能否反过来说出 ES 没事务、最终一致、不适合作为唯一存储（万一 ES 集群挂了，warrant 不能丢——这是法律文件）。
如果她答得好，可以滑到 sync 问题："那 MySQL 和 ES 之间怎么同步？"（这是天然的下一题，可以下次 session 再深入设计）。
Interviewer note — Core architectural question. Listen for: (1) clear role split (MySQL = source of truth, ES = search index), (2) why MySQL alone doesn't work (FULLTEXT/LIKE limits for fuzzy + multi-field + relevance), (3) why ES alone doesn't work (no transactions, eventual consistency, can't be sole store for legal documents). If she answers well, slide into the sync question.
📖 追问 1 面试官参考答案 / Follow-up 1 Interviewer Reference

🎯 核心定位 / Core role split

MySQL = 真理之源（source of truth） · Elasticsearch = 搜索索引副本
MySQL is the system of record. Elasticsearch is a search-optimized replica. Data is written to MySQL, then propagated to ES.

📊 维度对比 / Dimension comparison

需求 / NeedMySQL 适合ES 适合
事务、外键、强一致✅❌
按主键查一条（detail page）✅⚠️ 能但浪费
全文搜索（"含某关键词"）❌ FULLTEXT 简陋，LIKE '%x%' 慢✅ 倒排索引
多字段模糊过滤 + 相关性排序 + 高亮❌✅
聚合分析（group by 大数据量）⚠️ 慢✅
更新一行数据✅⚠️ 慢（要重建分词）
备份恢复 / 法律审计✅⚠️ 不规范

🚓 在 warrants workflow 里具体怎么落地 / Concrete mapping in the warrants context

  • MySQL 存什么：完整 warrant 记录（warrant_id, type, suspect_id, officer_id, judge_id, charges, status, probable_cause, created_at, ...）+ 外键关联到 case / suspect / officer / judge 表。法律文件不能丢、不能错。
    MySQL stores the full warrant record with FKs to case/suspect/officer/judge. Legal documents — can't be lost or wrong.
  • ES 存什么：搜索相关字段的拍平副本（warrant_number, suspect_name, charge_text, date, officer_name, status）+ 可能冗余几个详情字段。不是 source of truth，可重建。
    ES stores a flat, search-optimized projection (suspect name, charge text, status, etc.). Not source of truth — fully rebuildable from MySQL.
  • 典型查询场景：警察输入 "John Smith DUI 2020" → ES 模糊匹配 + 多字段 + 按相关性排序 → 毫秒级返回 → 点详情时再去 MySQL 拿完整数据。
    Officer searches "John Smith DUI 2020" → ES does fuzzy + multi-field + relevance ranking → click detail → fetch full record from MySQL.

❓ 为什么 MySQL 自己不够 / Why MySQL alone fails

  • LIKE '%keyword%' 不走索引，全表扫描，慢
  • MySQL FULLTEXT INDEX 存在但功能简陋：分词器弱、相关性打分原始、不支持高级查询 DSL（bool / nested / aggregation）、跨多字段配置麻烦
  • 没有原生的相关性排序，警察搜 "John Smith"，MySQL 不能自动按"最匹配的在前"返回
  • LIKE '%x%' = full table scan. MySQL FULLTEXT exists but is primitive — weak tokenizer, basic relevance scoring, no rich query DSL.

❓ 为什么 ES 自己也不够 / Why ES alone also fails

  • 没有事务：写两条 warrant 不能保证 atomic，warrants 这种法律文件不能容忍部分成功
  • 最终一致：默认 ~1s refresh interval，"写完立刻搜搜不到"的窗口期
  • 不规范的备份恢复：ES snapshot 流程比 MySQL 备份复杂，法律审计不喜欢
  • 没有外键 / 强 schema：warrant ↔ case ↔ officer 这种关系约束 ES 不强制
  • 集群挂了不能丢数据：如果 ES 是 source of truth，集群事故 = 全部 warrants 丢失，灾难
  • No transactions, eventual consistency (~1s refresh), no FKs/strict schema, recovery story weaker. If ES is sole store and the cluster dies, you lose legal documents — unacceptable.

🌊 同步机制（如果她答得好，可以滑到这层）/ Sync mechanism (optional next-level probe)

  • 双写：APP 同时写 MySQL + ES。简单但一致性脆弱（intern 项目大概率是这种）
  • MQ 解耦：APP → MySQL，发消息到 Kafka/RabbitMQ → Consumer → ES。生产推荐
  • CDC（Change Data Capture）：监听 MySQL binlog（Debezium / Canal），自动同步到 ES。最优雅但最复杂
  • 定时批处理：cron 定期抽数据写 ES。低端方案，实时性差
