🎯 核心定位 / Core role split

MySQL = 真理之源（source of truth） · Elasticsearch = 搜索索引副本
MySQL is the system of record. Elasticsearch is a search-optimized replica. Data is written to MySQL, then propagated to ES.

📊 维度对比 / Dimension comparison

需求 / NeedMySQL 适合ES 适合
事务、外键、强一致✅❌
按主键查一条（detail page）✅⚠️ 能但浪费

🚓 在 warrants workflow 里具体怎么落地 / Concrete mapping in the warrants context

  • MySQL 存什么：完整 warrant 记录（warrant_id, type, suspect_id, officer_id, judge_id, charges, status, probable_cause, created_at, ...）+ 外键关联到 case / suspect / officer / judge 表。法律文件不能丢、不能错。
    MySQL stores the full warrant record with FKs to case/suspect/officer/judge. Legal documents — can't be lost or wrong.
  • ES 存什么：搜索相关字段的拍平副本（warrant_number, suspect_name, charge_text, date, officer_name, status）+ 可能冗余几个详情字段。不是 source of truth，可重建。
    ES stores a flat, search-optimized projection (suspect name, charge text, status, etc.). Not source of truth — fully rebuildable from MySQL.
  • 典型查询场景：警察输入 "John Smith DUI 2020" → ES 模糊匹配 + 多字段 + 按相关性排序 → 毫秒级返回 → 点详情时再去 MySQL 拿完整数据。
    Officer searches "John Smith DUI 2020" → ES does fuzzy + multi-field + relevance ranking → click detail → fetch full record from MySQL.

❓ 为什么 MySQL 自己不够 / Why MySQL alone fails

  • LIKE '%keyword%' 不走索引，全表扫描，慢
  • MySQL FULLTEXT INDEX 存在但功能简陋：分词器弱、相关性打分原始、不支持高级查询 DSL（bool / nested / aggregation）、跨多字段配置麻烦
  • 没有原生的相关性排序，警察搜 "John Smith"，MySQL 不能自动按"最匹配的在前"返回
  • LIKE '%x%' = full table scan. MySQL FULLTEXT exists but is primitive — weak tokenizer, basic relevance scoring, no rich query DSL.

❓ 为什么 ES 自己也不够 / Why ES alone also fails

  • 没有事务：写两条 warrant 不能保证 atomic，warrants 这种法律文件不能容忍部分成功
  • 最终一致：默认 ~1s refresh interval，"写完立刻搜搜不到"的窗口期
  • 不规范的备份恢复：ES snapshot 流程比 MySQL 备份复杂，法律审计不喜欢
  • 没有外键 / 强 schema：warrant ↔ case ↔ officer 这种关系约束 ES 不强制
  • 集群挂了不能丢数据：如果 ES 是 source of truth，集群事故 = 全部 warrants 丢失，灾难
  • No transactions, eventual consistency (~1s refresh), no FKs/strict schema, recovery story weaker. If ES is sole store and the cluster dies, you lose legal documents — unacceptable.

🌊 同步机制（如果她答得好，可以滑到这层）/ Sync mechanism (optional next-level probe)

  • 双写：APP 同时写 MySQL + ES。简单但一致性脆弱（intern 项目大概率是这种）
  • MQ 解耦：APP → MySQL，发消息到 Kafka/RabbitMQ → Consumer → ES。生产推荐
  • CDC（Change Data Capture）：监听 MySQL binlog（Debezium / Canal），自动同步到 ES。最优雅但最复杂
  • 定时批处理：cron 定期抽数据写 ES。低端方案，实时性差

🌊 4 种主流方案 / 4 Mainstream Patterns

方案 / Pattern怎么做✅ 优点❌ 缺点
双写 / Double-Write Intern 最可能 APP 写 MySQL 后立即写 ES（或反之） 简单，无新组件 一致性脆弱—MySQL 成功但 ES 失败怎么办？无事务保证
MQ 解耦 生产推荐 APP → MySQL，发消息到 Kafka / RabbitMQ → Consumer → ES 失败可重试（消息留在 MQ）；可批量写 ES，性能好 引入 MQ 基础设施；最终一致；业务代码仍要发消息
CDC (Change Data Capture) 高级 · 加分 监听 MySQL binlog → Debezium / Canal → Kafka → Consumer → ES 业务代码零侵入；数据不会漏（binlog 是 SoT） 架构最复杂；运维成本高；同步延迟 ~秒级
定时批处理 / Batch 老派 cron 定时（按 updated_at）抽 MySQL 写 ES 实现最简单 实时性差（分钟～小时）；不适合用户面查询

  • 顶级实现：Debezium 监听 binlog，业务代码完全无感知

🛠 主流 CDC 工具 / Mainstream CDC tools

  • Canal — 阿里开源，老牌中文圈选择

▸ Step 1 · 写侧失败处理 / Write-side Failure Handling

  • 🟢 重试 + 指数退避：网络抖 / ES 临时挂的情况，自动重试通常能成
    Retry with exponential backoff for transient failures
  • 🟢 死信队列 (DLQ)：N 次重试仍失败的消息送入 DLQ，告警 + 人工 / 离线补偿
    Dead-letter queue for messages that exhaust retries
  • 🟢 监控告警：监控 ES 写入失败率、同步延迟；阈值告警
    Monitor write-failure rate and sync lag, alert on threshold breach
  • 🟢 对账 job (reconciliation)：定期跑批比对 MySQL 和 ES 的 count / checksum，发现 drift 报警 + 自动补数
    Reconciliation job — periodically compare counts/checksums, alert on drift, auto-repair
  • 🟢 顺序保证：用同一 partition key（如 warrant_id）保证同一条记录的更新按顺序消费；或带 version 字段 ES 端做 optimistic concurrency check
    Ordering — same partition key or version field with optimistic concurrency
  • 🚩 双写场景特别痛：用双写没事务保证 → 失败只能靠业务代码 try/catch 补偿；MQ/CDC 比双写更容易做可靠失败处理

▸ Step 2 · 读侧实时性 / Read-side Immediacy

  • 🎯 核心知识点 — ES refresh interval：ES 默认每 1s 把内存缓冲刷成新 segment 才能被搜到。所以 "写完立刻搜" 不保证能搜到。
    ES default refresh interval is 1s. "Write then immediately search" is NOT guaranteed to return.
  • 🟢 方式 1：强制 refresh — 调用时带 ?refresh=true 强制等到可搜。代价：写入吞吐严重下降（refresh 是重操作），只能用于关键场景
    Force refresh with ?refresh=true — guaranteed visible but kills write throughput
  • 🟢 方式 2：用 MySQL 兜底 — 关键场景 "写完立刻看" 不走 ES，直接查 MySQL（PK lookup 很快）
    Critical read-after-write paths go directly to MySQL (PK lookup is fast)
  • 🟢 方式 3：UX 绕过 — 警察刚创建 warrant 后跳详情页（按 ID 直查 MySQL）而非搜索页，自然避开 "搜不到" 窗口
    UX workaround — after create, route to detail page (PK fetch), not search
  • 🟢 方式 4：调短 refresh_interval — 改成 200ms 甚至更短，代价是 segment 碎片化 + merge 开销上涨
  • 🟡 更深的概念：ES 的 "near real-time" 是底层设计取舍 — 倒排索引 + segment 不可变 = 优秀的搜索性能，但写后立读窗口是必然代价

API缓存什么怎么用主要作用

▸ 进阶信号（说出任何一条 = senior）

  • useCallback 单独用毫无意义：如果下游子组件不是 React.memo'd，引用稳不稳定都会重渲，useCallback 等于白搭。
  • Memo 不是免费的：React.memo 每次 render 都要做浅比较；useMemo / useCallback 自身的依赖数组也要比较 + 函数本身要执行（即使没重算结果）。如果原本 render 很便宜，加 memo 反而更慢。
  • 默认应先调架构：能用组件拆分 / 状态下移 / children pattern 解决的 rerender，不要用 memo 兜底。Memo 是最后手段。
  • 依赖数组陷阱：
    - deps 写少了 → stale closure（函数闭包拿到旧 state / props）
    - deps 里放了不稳定引用（如每次 render 都是新对象）→ memo 立刻失效，等于没写
    - exhaustive-deps ESLint 规则是底线
  • "useMemo everywhere" 是反模式：到处套 memo 不仅性能没提升，还增加心智负担和 bug 面（依赖漏写）。
  • React Compiler / React 19（前沿 bonus）：编译器自动 memo，未来这三件套可能逐步淡出。

▸ Matching 算法决策流（高分答案的脉络） / Algorithm decision pipeline

  1. Email 规范化 / Normalize — 小写、去 +alias / .（Gmail 别名）、IDN 国际化域名
  2. 精确匹配 / Exact — WHERE contact.email = ?，命中即返回
  3. Thread 匹配 / Thread inheritance — 用 In-Reply-To / References 邮件头找历史邮件 → 继承上一封的 contact/deal 关联
  4. 域名匹配 / Domain — email.split('@')[1] 查 account.domain → 至少能挂到公司层
  5. 模糊匹配 / Fuzzy — display name / 历史 alias / Levenshtein / Jaro-Winkler
  6. 评分 + 阈值 / Score & threshold — > 高阈值 自动关联 / 中等 → 人工 review 队列 / 低 → unmatched 池
  7. 共享邮箱处理 / Shared inbox — sales@ support@ noreply@ 走特殊路径，不创建 contact
  8. 新 contact 策略 / New contact policy — 自动新建 vs 进 review 队列的判定规则
