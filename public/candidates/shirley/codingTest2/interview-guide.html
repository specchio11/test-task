📋 题面 / Problem Statement

English:

We're building front-end telemetry. When a user clicks the same button many times in a short window (a "rage-click"), it usually signals a perceived stall, slow loading, or interaction bug. Implement a small class that detects such bursts.

Implement ActivityBurstDetector:
  • Constructor (windowMs: number, threshold: number): window length in milliseconds, and the alarm threshold (a count).
  • Method hit(t: number): boolean: record one event at time t (in ms), then return whether — including this event — the number of events whose timestamps fall in the inclusive range [t - windowMs, t] is >= threshold.
Assumptions / 输入约束：
  • t is a non-negative integer (ms).
  • Successive calls to hit have strictly increasing t.
  • windowMs >= 0; threshold >= 1.
  • There is no unhit / clear. Events disappear only by aging out of the window.
Example: 
const d = new ActivityBurstDetector(1000, 3);
d.hit(100)   // false  — events in [-900, 100]:   [100]                count=1
d.hit(300)   // false  — events in [-700, 300]:   [100, 300]           count=2
d.hit(600)   // true   — events in [-400, 600]:   [100, 300, 600]      count=3 ✅
d.hit(2000)  // false  — events in [1000, 2000]:  [2000]               count=1  (100, 300, 600 evicted)
d.hit(2500)  // false  — events in [1500, 2500]:  [2000, 2500]         count=2
d.hit(2800)  // true   — events in [1800, 2800]:  [2000, 2500, 2800]   count=3 ✅

中文:

我们在做前端 telemetry。当用户在短时间内反复点同一个按钮（俗称 rage-click），通常意味着遇到了卡顿、loading、或交互 bug。请实现一个小工具类检测这种"爆发"。

实现 ActivityBurstDetector：
  • 构造函数 (windowMs: number, threshold: number)：窗口长度（毫秒）+ 触发阈值（次数）。
  • 方法 hit(t: number): boolean：在时刻 t（毫秒）记录一次事件，返回——包含本次在内——时间戳落在闭区间 [t - windowMs, t] 里的事件总数是否 >= threshold。
输入约束：
  • t 是非负整数（毫秒）。
  • hit 的连续调用，t 严格递增。
  • windowMs >= 0；threshold >= 1。
  • 没有 unhit / 清空操作。事件一旦记录，只能随时间走出窗口被自然淘汰。
示例： 
const d = new ActivityBurstDetector(1000, 3);
d.hit(100)   // false  — 窗口 [-900, 100]：  [100]                count=1
d.hit(300)   // false  — 窗口 [-700, 300]：  [100, 300]           count=2
d.hit(600)   // true   — 窗口 [-400, 600]：  [100, 300, 600]      count=3 ✅
d.hit(2000)  // false  — 窗口 [1000, 2000]： [2000]               count=1  （100/300/600 全被淘汰）
d.hit(2500)  // false  — 窗口 [1500, 2500]： [2000, 2500]         count=2
d.hit(2800)  // true   — 窗口 [1800, 2800]： [2000, 2500, 2800]   count=3 ✅

💡 标准答案 / Model Solution

核心思路 / Core Idea

中文：这是一道滑动窗口 + 队列淘汰的经典模型。题目给了三个关键的结构性礼物：(1) t 严格递增；(2) 窗口长度固定；(3) 没有 unhit。这三点合起来 ⇒ 任何事件一旦走出窗口就永远不会再相关 ⇒ 用 FIFO 队列维护"当前窗口内的事件"，新事件入队、旧事件出队，每次询问只需读队列 size 与 threshold 比较。摊还复杂度 O(1) per hit。

English: Classic sliding-window with queue eviction. The problem gives three structural gifts: (1) t is strictly increasing; (2) window length is fixed; (3) no unhit. Together they imply: once an event ages out of the window it can never become relevant again. So maintain a FIFO queue of "events currently in the window" — enqueue on new events, dequeue old ones from the head. Each query is just size >= threshold. Amortized O(1) per hit.

关键观察 / Key Observations

  1. 窗口右端单调右移：由于 t 严格递增，每次 hit 时窗口右端就是当前 t，永远不回退。
    Window's right edge advances monotonically because t is strictly increasing.
  2. 窗口左端也单调右移：左端 = t - windowMs，windowMs 是常数，所以 t 涨 ⇒ 左端涨。⇒ 队首元素一旦小于左端，就再也不会回到窗口里，可以放心丢弃。
    Left edge t - windowMs also advances monotonically. So any front element below the left edge can be safely dropped — it will never re-enter.
  3. 为什么不需要二分：直觉上"找最早一个 ≥ cutoff 的位置"可以 binary search，O(log n)。但因为左端单调推进，用指针顺序淘汰是摊还 O(1)——每个事件一生总成本 = 1 次入队 + 1 次淘汰 = O(1)。
    Could use binary search (O(log n)), but linear front-eviction is amortized O(1) — each event is enqueued once and evicted once across its lifetime.
  4. 不变式：hit(t) 结束时，队列里所有时间戳都在 [t - windowMs, t] 内、且按时间升序。所以 size >= threshold 就是答案。
    Invariant: after hit(t) returns, all timestamps in the queue lie in [t - windowMs, t] in ascending order — so size >= threshold directly answers the query.

算法流程 / Algorithm Step-by-Step

  1. 第一步：维护一个 FIFO 队列 events（C++/Python 用 deque；C# 用 Queue；JS/TS 用普通数组 + head pointer，理由见下）。
  2. 第二步：调用 hit(t)：
    • 把 t 入队到队尾。
    • 计算 cutoff = t - windowMs。
    • 当队首 < cutoff 时反复出队，直到队首 >= cutoff 或队空。
    • 返回 队列 size >= threshold。
🚨 JS / TS 候选人特别注意：
Array.prototype.shift() 在 V8 里是 O(n)（数组要 reindex），会让摊还分析失效。正确做法是用一个 head 索引在数组上做"逻辑出队"，配合周期性 slice 压缩。详见下方 TypeScript 参考答案。
In V8, Array.prototype.shift() is O(n) (reindex), breaking the amortized analysis. Use a head pointer + periodic compaction instead — see the TypeScript reference.

代码 / Reference Code (4 个语言 · tab 切换 · 默认 C++)

四个版本都是同一份算法（FIFO 队列 + while-evict），差别仅在队列结构怎么实现：
All four use the same algorithm (FIFO queue + while-evict); only the queue implementation differs:

  • C++：std::deque<int>，push_back / pop_front 都是 O(1)。
  • C#：Queue<int>，Enqueue / Dequeue 都是摊还 O(1)。
  • Python：collections.deque，append / popleft 都是 O(1)。
  • TypeScript：普通数组 + head 索引（因为 .shift() 是 O(n)）；周期性 slice 压缩防止数组无限增长。

C++ 参考答案 C++17

#include <deque>

class ActivityBurstDetector {
public:
    ActivityBurstDetector(int windowMs, int threshold)
        : windowMs_(windowMs), threshold_(threshold) {}

    bool hit(int t) {
        events_.push_back(t);
        const int cutoff = t - windowMs_;
        while (!events_.empty() && events_.front() < cutoff) {
            events_.pop_front();
        }
        return static_cast<int>(events_.size()) >= threshold_;
    }

private:
    int windowMs_;
    int threshold_;
    std::deque<int> events_;   // timestamps currently inside the window
};

std::deque 的 push_back 和 pop_front 都是 O(1)。摊还每次 hit 是 O(1)。

C# 参考答案 C# / .NET 6+

using System.Collections.Generic;

public class ActivityBurstDetector {
    private readonly int windowMs;
    private readonly int threshold;
    private readonly Queue<int> events = new Queue<int>();

    public ActivityBurstDetector(int windowMs, int threshold) {
        this.windowMs = windowMs;
        this.threshold = threshold;
    }

    public bool Hit(int t) {
        events.Enqueue(t);
        int cutoff = t - windowMs;
        while (events.Count > 0 && events.Peek() < cutoff) {
            events.Dequeue();
        }
        return events.Count >= threshold;
    }
}

Queue<T> 内部是循环数组，Enqueue / Dequeue 摊还 O(1)。

Python 参考答案 Python 3

from collections import deque

class ActivityBurstDetector:
    def __init__(self, window_ms: int, threshold: int):
        self.window_ms = window_ms
        self.threshold = threshold
        self.events: deque[int] = deque()    # timestamps inside the window

    def hit(self, t: int) -> bool:
        self.events.append(t)
        cutoff = t - self.window_ms
        while self.events and self.events[0] < cutoff:
            self.events.popleft()
        return len(self.events) >= self.threshold

collections.deque 是 doubly-linked block list，append / popleft 都是真正的 O(1)（不像 list.pop(0) 是 O(n)）。

TypeScript 参考答案 TypeScript

class ActivityBurstDetector {
    private readonly windowMs: number;
    private readonly threshold: number;
    private events: number[] = [];
    private head = 0;   // logical front of the queue

    constructor(windowMs: number, threshold: number) {
        this.windowMs = windowMs;
        this.threshold = threshold;
    }

    hit(t: number): boolean {
        this.events.push(t);
        const cutoff = t - this.windowMs;
        while (
            this.head < this.events.length &&
            this.events[this.head] < cutoff
        ) {
            this.head++;
        }
        // Periodically compact so the underlying array doesn't grow forever.
        if (this.head > 1024 && this.head * 2 > this.events.length) {
            this.events = this.events.slice(this.head);
            this.head = 0;
        }
        return this.events.length - this.head >= this.threshold;
    }
}

为什么不直接 Array.prototype.shift()？V8 把 shift 实现成 O(n)（所有元素的索引要前移），会让一次 hit 退化成 O(n)、摊还分析也失效。用 head 索引做"逻辑出队"，再加一个周期性 slice 压缩防止底层数组无限增长，就能拿回真正的摊还 O(1)。

复杂度 / Complexity

  • 时间：每次 hit 摊还 O(1)。最坏单次可能 O(k)（k = 一次性淘汰多少老事件），但每个事件一生只会被入队/淘汰各一次，n 次 hit 总成本 O(n)，所以平均 O(1)/op。
  • 空间：O(w)，其中 w = 窗口内同时存在的事件数上界（与 hit 调用频率 × windowMs 相关，而非总调用次数）。
