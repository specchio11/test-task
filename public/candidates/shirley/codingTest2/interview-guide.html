📋 题面 / Problem Statement

English:

We're building front-end telemetry. When a user clicks the same button many times in a short window (a "rage-click"), it usually signals a perceived stall, slow loading, or interaction bug. Implement a small class that detects such bursts.

Implement ActivityBurstDetector:
  • Constructor (windowMs: number, threshold: number): window length in milliseconds, and the alarm threshold (a count).
  • Method hit(t: number): boolean: record one event at time t (in ms), then return whether — including this event — the number of events whose timestamps fall in the inclusive range [t - windowMs, t] is >= threshold.
Assumptions / 输入约束：
  • t is a non-negative integer (ms).
  • Successive calls to hit have strictly increasing t.
  • windowMs >= 0; threshold >= 1.
  • There is no unhit / clear. Events disappear only by aging out of the window.
Example: 
const d = new ActivityBurstDetector(1000, 3);
d.hit(100)   // false  — events in [-900, 100]:   [100]                count=1
d.hit(300)   // false  — events in [-700, 300]:   [100, 300]           count=2
d.hit(600)   // true   — events in [-400, 600]:   [100, 300, 600]      count=3 ✅
d.hit(2000)  // false  — events in [1000, 2000]:  [2000]               count=1  (100, 300, 600 evicted)
d.hit(2500)  // false  — events in [1500, 2500]:  [2000, 2500]         count=2
d.hit(2800)  // true   — events in [1800, 2800]:  [2000, 2500, 2800]   count=3 ✅

中文:

我们在做前端 telemetry。当用户在短时间内反复点同一个按钮（俗称 rage-click），通常意味着遇到了卡顿、loading、或交互 bug。请实现一个小工具类检测这种"爆发"。

实现 ActivityBurstDetector：
  • 构造函数 (windowMs: number, threshold: number)：窗口长度（毫秒）+ 触发阈值（次数）。
  • 方法 hit(t: number): boolean：在时刻 t（毫秒）记录一次事件，返回——包含本次在内——时间戳落在闭区间 [t - windowMs, t] 里的事件总数是否 >= threshold。
输入约束：
  • t 是非负整数（毫秒）。
  • hit 的连续调用，t 严格递增。
  • windowMs >= 0；threshold >= 1。
  • 没有 unhit / 清空操作。事件一旦记录，只能随时间走出窗口被自然淘汰。
示例： 
const d = new ActivityBurstDetector(1000, 3);
d.hit(100)   // false  — 窗口 [-900, 100]：  [100]                count=1
d.hit(300)   // false  — 窗口 [-700, 300]：  [100, 300]           count=2
d.hit(600)   // true   — 窗口 [-400, 600]：  [100, 300, 600]      count=3 ✅
d.hit(2000)  // false  — 窗口 [1000, 2000]： [2000]               count=1  （100/300/600 全被淘汰）
d.hit(2500)  // false  — 窗口 [1500, 2500]： [2000, 2500]         count=2
d.hit(2800)  // true   — 窗口 [1800, 2800]： [2000, 2500, 2800]   count=3 ✅

🗣️ 推荐面试时直接用这个版本念给候选人

English:

"Alright, here's a small library you're going to build for our front-end telemetry. Picture this: there's a button the user keeps clicking — when they hit it too many times in a short window, we want to know about it, because that usually means they ran into a bug or a slow load.

So you write a class called ActivityBurstDetector. The constructor takes two things: a window length in milliseconds and a threshold count. Then it has one method, hit(t), which you call every time the user does the thing — you pass in the current timestamp in ms, and the method returns a boolean: counting this one, are there now at least threshold events in the last windowMs milliseconds?

A few things to make life easy: timestamps come in strictly increasing order, so no out-of-order events. There's no unhit — once an event is too old, it just naturally falls out of the window. windowMs and threshold stay fixed for the lifetime of one detector. Make sense?"

中文:

"好，给你一个小工具类要实现，是给我们前端 telemetry 用的。场景是这样：有一个按钮用户在一直点——如果短时间内点了很多次，我们就想知道，因为这通常意味着他遇到了 bug 或者加载卡住了。

你写一个类叫 ActivityBurstDetector，构造函数接两个参数：窗口长度（毫秒）+ 触发阈值（次数）。然后有一个方法叫 hit(t)，用户每做一次行为你就调一次，传入当前时间戳（毫秒），方法返回一个 boolean：把本次也算上，最近 windowMs 毫秒里是不是累计达到 threshold 次。

几个简化条件：时间戳严格递增，不会乱序。没有 unhit / 撤销——事件一旦太老就自然滑出窗口。windowMs 和 threshold 在 detector 整个生命周期里都是固定的。听懂了吗？"

🧸 当候选人对"窗口 / 阈值"这种描述觉得抽象时使用

English:

"Let me strip the jargon. Picture a front desk. Every time someone walks up, you jot down the arrival time on a notepad.

The rule is: if 3 or more people walk up within any 60-second window, ring the alarm. So whenever someone arrives, you (1) write down the time, (2) glance at the notepad — within the last 60 seconds (including the one you just wrote), are there at least 3 timestamps? If yes, ring the alarm and return true; otherwise return false.

People only arrive in chronological order — you'll never get an out-of-order arrival. And timestamps older than 60 seconds you can just erase — they're never going to matter again.

That's the whole job: a class with constructor (window, threshold) — in the example above 60 and 3 — and one method hit(t) that returns whether the alarm fires."

中文:

"我换个最简单的说法。想象你坐在前台。每来一个客人，你就在便签上记下到达时间。

规定是：最近 60 秒内只要来了 3 个或更多人，就敲警报。所以每来一个客人，你 (1) 记下时间；(2) 看一眼便签——最近 60 秒内（包括刚记的这次）是不是至少有 3 个时间？是就敲警报、返回 true；否则返回 false。

客人按时间先后来，不会乱序。超出 60 秒的旧时间擦掉就行——以后永远用不到了。

就这么简单：写一个类，构造函数 (window, threshold)——上面例子里就是 60 和 3——一个方法 hit(t) 返回警报是否触发。"

🚨 候选人卡壳情况及提示策略 / Sticking Points & Hints

🤔 卡壳 0：不理解题目术语 / Stuck 0: Doesn't understand the terminology

表现 / Symptom：对 "window"、"threshold"、"strictly increasing"、"hit" 的含义反复确认；或把 windowMs 当成"未来还允许多少毫秒"而不是"过去多长的窗口"。

解释 "window" / Explaining "window"：

"windowMs is a fixed look-back length. When hit(t) is called, the relevant time range is [t − windowMs, t], a closed interval. Events older than t − windowMs are out of the window and do not count."

"windowMs 是一个固定往回看的长度。hit(t) 被调用时，相关的时间区间是 [t − windowMs, t]，闭区间。早于 t − windowMs 的事件已经走出窗口，不算。"

解释 "threshold" / Explaining "threshold"：

"Return true if and only if the count of events inside [t − windowMs, t] — including the one just inserted by this call — is >= threshold."

"当且仅当 [t − windowMs, t] 区间内的事件数（含本次刚插入的这个）>= threshold 时返回 true。"

解释 "strictly increasing t" / Explaining "strictly increasing t"：

"Every call to hit has a t strictly larger than all previous t's. You will never see out-of-order or repeated timestamps. This is a real-world simplification — events arrive in the order they happen."

"每次 hit 传入的 t 都严格大于之前所有的 t。不会出现乱序或重复时间戳。这是一个现实简化——事件按发生顺序到达。"

用样例帮助理解 / Walking through the sample：

"Look at ActivityBurstDetector(1000, 3). After hit(100) and hit(300), both returned false — only 1 then 2 events in window. hit(600) returns true because [100, 300, 600] are all within [−400, 600] — count is 3, meets threshold. Then hit(2000): the window is now [1000, 2000], so 100/300/600 are all evicted, only [2000] remains, count = 1."

"看 ActivityBurstDetector(1000, 3)。hit(100)、hit(300) 都返回 false——窗口里只有 1 / 2 个事件。hit(600) 返回 true，因为 [100, 300, 600] 都在 [−400, 600] 范围内，count = 3 达到 threshold。然后 hit(2000)：窗口已变成 [1000, 2000]，100/300/600 全部走出窗口，只剩 [2000]，count = 1。"
⚠️ 面试官注意：澄清术语不算提示，不应影响算法能力评分。术语对非英语母语候选人是公平的解释场景。但如果理解题意后仍长时间没思路，按卡壳 1 处理。
Clarifying terminology is not a hint and should not affect the algorithm-skill evaluation. If the candidate still has no idea after understanding the problem, treat as Stuck 1.
😶 卡壳 1：完全没思路 / 想立刻用 binary search / Stuck 1: No idea, or jumps to binary search

表现 / Symptom：沉默；或者说 "I'll store all timestamps in an array and binary-search the left boundary every call"——能算出答案但忽视了"事件走出窗口后永远不会回来"这个结构性礼物，复杂度会停在 O(log n) per hit。

提示等级English中文
Light Hint 1 "What state do you need to keep between calls? After hit(t) returns, what information from past events do you still need for the next call?" "你需要在两次调用之间维护什么状态？hit(t) 返回后，过去的事件里哪些信息对下次调用还有用？"
Light Hint 2 "t is strictly increasing. Once an event is older than t − windowMs for the current call, can it ever become relevant again in any future call?" "t 严格递增。一个事件一旦在当前调用里比 t − windowMs 还旧了，未来还有可能重新进入窗口吗？"
Medium Hint 3 "Maintain a list of 'events currently inside the window'. On each hit(t): append the new one, then trim. What kind of data structure makes the trim cheap?" "维护一个"当前窗口内事件"的列表。每次 hit(t)：先放入新事件，再修剪窗口。什么样的数据结构能让修剪很便宜？"
Heavy Hint 4 "Use a FIFO queue. On hit(t): enqueue(t); while front < t − windowMs, dequeue; return size() >= threshold. Each event is enqueued once, dequeued once — amortized O(1) per call." "用 FIFO 队列。hit(t)：enqueue(t)；当队首 < t − windowMs 时反复 dequeue；返回 size() >= threshold。每个事件一生只入队一次、出队一次——摊还 O(1) per call。"
🐌 卡壳 2：每次 hit 全量扫一遍 / Stuck 2: Linear scan on every call

表现 / Symptom：用 list / vector 存所有事件，每次 hit 都 filter / 重新遍历计数。功能上正确但复杂度 O(n) per hit、空间 O(N total)。在 followup 复杂度环节就会暴露。

提示等级English中文
Light Hint 1 "What's the time complexity of one hit call in your current code? What about total work over N calls?" "你现在的 hit 一次调用是什么复杂度？N 次调用累计呢？"
Medium Hint 2 "You said events older than t − windowMs never re-enter the window. Are you keeping them around in memory? Why?" "你之前说过比 t − windowMs 旧的事件永远不会回到窗口。那你为什么还把它们留在内存里？"
Heavy Hint 3 "Replace the filter with eviction: keep events in arrival order; on each hit(t), evict from the front while front < t − windowMs. Each event is touched O(1) times across its lifetime." "把 filter 换成 evict：事件按到达顺序保存；每次 hit(t) 从队首循环淘汰过期的。每个事件一生只被处理 O(1) 次。"
⚠️ 面试官注意：这一卡壳不一定会发生——很多候选人会直接想到队列。如果直接跳到正确解就跳过这一节。但如果给了 Heavy hint 还卡在"全量扫"或者写不出 while-loop 淘汰，按卡壳 3 排查具体的数据结构选择。
Skip this section if the candidate jumps straight to a queue. Drop to Stuck 3 if they still can't articulate the eviction loop after Heavy hint.
🪤 卡壳 3：数据结构选错（语言陷阱）/ Stuck 3: Wrong data structure for O(1) front-removal (language pitfall)

表现 / Symptom：算法思路正确（队列 + while-evict），但用了一个O(n) 队首删除的结构：
• JS / TS：用 Array.prototype.shift()（V8 上是 O(n)，需移动所有元素）
• Python：用 list + list.pop(0)（同样 O(n)）
• C#：用 List<int>.RemoveAt(0)（O(n)，应该用 Queue<int>）
• C++：通常没问题（多数人会自然选 std::deque 或 std::queue），但偶尔会用 std::vector + erase(begin())（O(n)）。

这是本题对 Shirley 最关键的一关。她是 JS/TS 重度用户，能否意识到 shift() 的代价、并主动选择 head-pointer / Deque-like 结构，是判断她"对 V8 性能模型有多熟"的强信号。

提示等级English中文
Light Hint 1 "You wrote events.shift() [or events.pop(0)]. What's the cost of removing from the front in this language's array/list? Is it really O(1)?" "你写了 events.shift()（或 events.pop(0)）。这门语言的数组/列表从头部删元素是什么复杂度？真的是 O(1) 吗？"
Medium Hint 2 "In V8, shift() moves every remaining element down by one slot — that's O(n). The amortized O(1) analysis you'd hoped for is gone. What other structures support O(1) front-removal in this language?" "V8 里 shift() 会把剩余元素逐个左移——O(n)。你期望的摊还 O(1) 没了。这门语言里还有什么数据结构支持 O(1) 头部删除？"
Heavy Hint 3 "Per-language fixes:
• JS/TS: keep a head pointer — let head = 0; while (events[head] < cutoff) head++; and count is events.length − head. Or build a tiny Deque class backed by a circular buffer.
• Python: from collections import deque; events.popleft().
• C#: Queue<int> with Dequeue() / Count.
• C++: std::deque<int> with pop_front()." 		"各语言对应改法：
• JS/TS：用 head pointer——let head = 0; while (events[head] < cutoff) head++;，count = events.length − head。或封装一个用环形 buffer 的 Deque 类。
• Python：from collections import deque; events.popleft()。
• C#：Queue<int>，用 Dequeue() / Count。
• C++：std::deque<int>，用 pop_front()。"
⚠️ 面试官注意：如果候选人主动避开 shift / pop(0) 并解释原因（"shift 是 O(n) 因为要左移所有元素"），这是非常强的语言深度信号，可直接归到节点 3 的 🌟 评级，省掉 Hint 1。
If the candidate proactively avoids shift / pop(0) and explains why, that's a very strong language-depth signal — mark Checkpoint 3 as 🌟 and skip Hint 1.

节点 4：摊还复杂度分析 / Amortized complexity analysis

期望反应 / Expected：主动给出 worst-case O(n) per call vs amortized O(1) per call 的双视角分析，并解释 amortization 的来源：每个事件一生总成本 = 1 次入队 + 至多 1 次出队 = O(1)，N 次 hit 总共 O(N)。空间 O(W)，W = 窗口内事件数上界。

💡 标准答案 / Model Solution

核心思路 / Core Idea

中文：这是一道滑动窗口 + 队列淘汰的经典模型。题目给了三个关键的结构性礼物：(1) t 严格递增；(2) 窗口长度固定；(3) 没有 unhit。这三点合起来 ⇒ 任何事件一旦走出窗口就永远不会再相关 ⇒ 用 FIFO 队列维护"当前窗口内的事件"，新事件入队、旧事件出队，每次询问只需读队列 size 与 threshold 比较。摊还复杂度 O(1) per hit。

English: Classic sliding-window with queue eviction. The problem gives three structural gifts: (1) t is strictly increasing; (2) window length is fixed; (3) no unhit. Together they imply: once an event ages out of the window it can never become relevant again. So maintain a FIFO queue of "events currently in the window" — enqueue on new events, dequeue old ones from the head. Each query is just size >= threshold. Amortized O(1) per hit.

关键观察 / Key Observations

  1. 窗口右端单调右移：由于 t 严格递增，每次 hit 时窗口右端就是当前 t，永远不回退。
    Window's right edge advances monotonically because t is strictly increasing.
  2. 窗口左端也单调右移：左端 = t - windowMs，windowMs 是常数，所以 t 涨 ⇒ 左端涨。⇒ 队首元素一旦小于左端，就再也不会回到窗口里，可以放心丢弃。
    Left edge t - windowMs also advances monotonically. So any front element below the left edge can be safely dropped — it will never re-enter.
  3. 为什么不需要二分：直觉上"找最早一个 ≥ cutoff 的位置"可以 binary search，O(log n)。但因为左端单调推进，用指针顺序淘汰是摊还 O(1)——每个事件一生总成本 = 1 次入队 + 1 次淘汰 = O(1)。
    Could use binary search (O(log n)), but linear front-eviction is amortized O(1) — each event is enqueued once and evicted once across its lifetime.
  4. 不变式：hit(t) 结束时，队列里所有时间戳都在 [t - windowMs, t] 内、且按时间升序。所以 size >= threshold 就是答案。
    Invariant: after hit(t) returns, all timestamps in the queue lie in [t - windowMs, t] in ascending order — so size >= threshold directly answers the query.

算法流程 / Algorithm Step-by-Step

  1. 第一步：维护一个 FIFO 队列 events（C++/Python 用 deque；C# 用 Queue；JS/TS 用普通数组 + head pointer，理由见下）。
  2. 第二步：调用 hit(t)：
    • 把 t 入队到队尾。
    • 计算 cutoff = t - windowMs。
    • 当队首 < cutoff 时反复出队，直到队首 >= cutoff 或队空。
    • 返回 队列 size >= threshold。
🚨 JS / TS 候选人特别注意：
Array.prototype.shift() 在 V8 里是 O(n)（数组要 reindex），会让摊还分析失效。正确做法是用一个 head 索引在数组上做"逻辑出队"，配合周期性 slice 压缩。详见下方 TypeScript 参考答案。
In V8, Array.prototype.shift() is O(n) (reindex), breaking the amortized analysis. Use a head pointer + periodic compaction instead — see the TypeScript reference.

代码 / Reference Code (4 个语言 · tab 切换 · 默认 C++)

四个版本都是同一份算法（FIFO 队列 + while-evict），差别仅在队列结构怎么实现：
All four use the same algorithm (FIFO queue + while-evict); only the queue implementation differs:

  • C++：std::deque<int>，push_back / pop_front 都是 O(1)。
  • C#：Queue<int>，Enqueue / Dequeue 都是摊还 O(1)。
  • Python：collections.deque，append / popleft 都是 O(1)。
  • TypeScript：普通数组 + head 索引（因为 .shift() 是 O(n)）；周期性 slice 压缩防止数组无限增长。

C++ 参考答案 C++17

#include <deque>

class ActivityBurstDetector {
public:
    ActivityBurstDetector(int windowMs, int threshold)
        : windowMs_(windowMs), threshold_(threshold) {}

    bool hit(int t) {
        events_.push_back(t);
        const int cutoff = t - windowMs_;
        while (!events_.empty() && events_.front() < cutoff) {
            events_.pop_front();
        }
        return static_cast<int>(events_.size()) >= threshold_;
    }

private:
    int windowMs_;
    int threshold_;
    std::deque<int> events_;   // timestamps currently inside the window
};

std::deque 的 push_back 和 pop_front 都是 O(1)。摊还每次 hit 是 O(1)。

C# 参考答案 C# / .NET 6+

using System.Collections.Generic;

public class ActivityBurstDetector {
    private readonly int windowMs;
    private readonly int threshold;
    private readonly Queue<int> events = new Queue<int>();

    public ActivityBurstDetector(int windowMs, int threshold) {
        this.windowMs = windowMs;
        this.threshold = threshold;
    }

    public bool Hit(int t) {
        events.Enqueue(t);
        int cutoff = t - windowMs;
        while (events.Count > 0 && events.Peek() < cutoff) {
            events.Dequeue();
        }
        return events.Count >= threshold;
    }
}

Queue<T> 内部是循环数组，Enqueue / Dequeue 摊还 O(1)。

Python 参考答案 Python 3

from collections import deque

class ActivityBurstDetector:
    def __init__(self, window_ms: int, threshold: int):
        self.window_ms = window_ms
        self.threshold = threshold
        self.events: deque[int] = deque()    # timestamps inside the window

    def hit(self, t: int) -> bool:
        self.events.append(t)
        cutoff = t - self.window_ms
        while self.events and self.events[0] < cutoff:
            self.events.popleft()
        return len(self.events) >= self.threshold

collections.deque 是 doubly-linked block list，append / popleft 都是真正的 O(1)（不像 list.pop(0) 是 O(n)）。

TypeScript 参考答案 TypeScript

class ActivityBurstDetector {
    private readonly windowMs: number;
    private readonly threshold: number;
    private events: number[] = [];
    private head = 0;   // logical front of the queue

    constructor(windowMs: number, threshold: number) {
        this.windowMs = windowMs;
        this.threshold = threshold;
    }

    hit(t: number): boolean {
        this.events.push(t);
        const cutoff = t - this.windowMs;
        while (
            this.head < this.events.length &&
            this.events[this.head] < cutoff
        ) {
            this.head++;
        }
        // Periodically compact so the underlying array doesn't grow forever.
        if (this.head > 1024 && this.head * 2 > this.events.length) {
            this.events = this.events.slice(this.head);
            this.head = 0;
        }
        return this.events.length - this.head >= this.threshold;
    }
}

为什么不直接 Array.prototype.shift()？V8 把 shift 实现成 O(n)（所有元素的索引要前移），会让一次 hit 退化成 O(n)、摊还分析也失效。用 head 索引做"逻辑出队"，再加一个周期性 slice 压缩防止底层数组无限增长，就能拿回真正的摊还 O(1)。

复杂度 / Complexity

  • 时间：每次 hit 摊还 O(1)。最坏单次可能 O(k)（k = 一次性淘汰多少老事件），但每个事件一生只会被入队/淘汰各一次，n 次 hit 总成本 O(n)，所以平均 O(1)/op。
  • 空间：O(w)，其中 w = 窗口内同时存在的事件数上界（与 hit 调用频率 × windowMs 相关，而非总调用次数）。
