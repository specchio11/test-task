📌 简历原文 / Resume excerpt — Amazon Web Services · Vancouver, BC · Software Development Engineer II (May 2025 – Present), Bullet ①:

"Expanded Backup Audit Manager (BAM) cross-region, cross-account support to 10 opt-in global regions."

"将 Backup Audit Manager (BAM) 的跨 region、跨账号支持扩展到了 10 个 opt-in 全球 region。"

🎯 考察意图 / Probing intent：这是 5 条 bullet 里信号最高的一条——含 3 个具体可验证的名词（BAM / cross-region + cross-account / opt-in regions）。从"业务为啥需要"一路逼到"region 6 → 7 之间的 safety check 是什么"，验证她在 AWS SDE II 一年内是真的端到端 own 过这个扩展，还是只参与了一部分。
This is the highest-signal bullet of the 5 — it contains 3 verifiable concrete terms (BAM / cross-region+cross-account / opt-in regions). Drill from "why customers need it" down to "what's your safety check between region 6 and region 7", verifying whether she actually owned this end-to-end in her first year at AWS or just contributed to a piece.

🎤 主问题 / Main Question — 什么是 BAM / What is BAM

🎤 提问 / Ask
"Before we dive into the expansion work — for someone who hasn't used Backup Audit Manager, can you give me a quick intro? What is BAM, what problem does it solve, and who are the typical customers using it?"
"在聊扩展工作之前——先介绍一下，对于没用过 Backup Audit Manager 的人，BAM 是什么？它解决的是什么问题？典型客户是谁？"

🎯 期望听到的内容 / What you want to hear:

  • BAM 在 AWS Backup 服务家族下，专门做合规 / 审计层 — 让客户能定义 backup frameworks（一组规则），自动持续审计哪些 resource 是否满足备份策略 / BAM sits under AWS Backup as the compliance / audit layer — customers define backup frameworks (rule sets) and BAM continuously audits resources against them
  • 明确的客户画像：受监管行业（金融、医疗、政府）的 compliance team，或者大企业的 governance / FinOps 团队 / Clear customer profile: regulated industries (financial, healthcare, government) compliance teams; enterprise governance / FinOps teams
  • 能讲出 cross-region + cross-account 为什么对这个服务特别关键：审计本身就是跨整个 organization 看的，单 region / 单 account 的审计在 enterprise 没意义 / Can articulate why cross-region + cross-account is core to this service: audit by nature spans the whole org; single-region / single-account audit is meaningless at enterprise scale
  • 知道至少一个具体的合规框架名字（HIPAA / FedRAMP / SOC2 / PCI），或 AWS Backup framework 里的 control 例子 / Names at least one specific compliance framework (HIPAA / FedRAMP / SOC2 / PCI) or an example AWS Backup framework control

🚩 红旗信号 / Red flags:

  • "BAM 就是 backup 的 audit" — 同义反复，没解释清 audit 在这里具体是什么 / "BAM is the audit for backup" — tautology with no explanation
  • 说不出客户长什么样，只能讲 "enterprise customers" / Can't describe customer profile beyond "enterprise customers"
  • 把 BAM 跟 AWS Backup、AWS Config、CloudTrail 搞混 — 这些在 audit 链路里各司其职，分不清就是没真做过 / Confuses BAM with AWS Backup / AWS Config / CloudTrail — each plays a different role in the audit chain, mixing them up = hasn't done the work

💡 如果她讲得太抽象 / If she stays abstract:

"Give me one concrete example — a customer turns BAM on for their AWS organization. What does BAM start doing on day one?"
"举一个具体例子——客户在自己的 AWS organization 开了 BAM，第一天 BAM 开始干什么？"

↪ 追问 1 / Follow-up 1 — 为什么 expand + 怎么实现 / Why Expand & How It Was Built

🎤 提问 / Ask
"OK, so why did BAM need to expand cross-region, cross-account support to these 10 opt-in regions? What was the driver — customer demand, a compliance deadline, a regional GA? And then walk me through how the expansion was actually implemented — what changed in the code / infra / config to make a new region 'work'?"
"好——那为什么 BAM 要把 cross-region、cross-account support 扩展到这 10 个 opt-in region？驱动是什么——客户需求？合规 deadline？还是这些 region 刚 GA？然后，给我讲讲这个扩展具体怎么实现的——代码 / infra / config 里改了什么，才让一个新 region "能用"？"

🎯 期望听到的内容 / What you want to hear:

  • 明确的驱动：客户实际需求（哪类客户、什么 use case）/ 监管要求 / 这批 opt-in region 自身是逐步 GA 的 / Clear driver: specific customer ask, regulatory deadline, or these opt-in regions themselves recently went GA
  • 知道 opt-in region 跟普通 region 的本质差别：默认关闭（用户必须主动 enable）/ STS session token 默认不通用 / SDK endpoint 不会自动解析 / 每个 region 有独立的 service principal / Knows the actual differences: disabled by default; STS session tokens not portable; SDK endpoints don't auto-resolve; each region has its own service principal
  • 实现侧讲清"region onboarding"涉及哪几层：(a) service 在 region 起一组新 stack / cell；(b) IAM role / trust policy 新增该 region；(c) 跨 region 调用时的 endpoint resolution / credential 跨域；(d) data residency—审计数据是 region-local 还是 home-region 聚合？/ Implementation covers: (a) deploying a new stack/cell in the region; (b) updating IAM roles / trust policies; (c) endpoint resolution + credentials across regions; (d) data residency — is audit data region-local or aggregated to a home region?
  • 能给出至少一个"扩到这个 region 时我们特别撞到的坑"——比如某个 opt-in region 不支持某个 dependency service、或者 STS endpoint 的 fallback 行为 / Names at least one specific gotcha encountered ("when we extended to region X we hit Y")

🚩 红旗信号 / Red flags:

  • "客户要的" / "PM 让做的" 不带任何 use case 细节 / Vague "customer asked" / "PM scoped it" with no use-case detail
  • 不知道 opt-in region 跟普通 region 有啥区别 — 简历字面就有 "opt-in"，答不出意味着 bullet 是包装 / Doesn't know what opt-in region means — when the bullet literally says "opt-in", this means the bullet is padded
  • 实现讲成 "我们把 region list 加了 10 个 entry" — 把它当 config flag，不知道 region onboarding 是个系统工程 / Reduces implementation to "we added 10 entries to a region list" — treats it as a config flag, doesn't realize region onboarding is a systems undertaking
  • 说不出 cross-region 调用时 credential / endpoint 怎么处理 — cross-region 工作核心问题之一 / Can't articulate cross-region credential / endpoint handling — one of the core problems

💡 如果她卡壳 / If stuck — 给一个起点 / Anchor:

"Let's pick one specific region — say ap-east-1 (Hong Kong) or me-south-1 (Bahrain). What was on your checklist before you could ship 'BAM works here'?"
"我们挑一个具体 region——比如 ap-east-1（香港）或者 me-south-1（巴林）。在你能说"BAM 在这里能用了" 之前，你的 checklist 上有什么？"

↪ 追问 2 / Follow-up 2 — Partial failure 契约 / Partial Failure Contract 🌟 Senior 指示器

🎤 提问 / Ask
"BAM is a cross-region aggregation service — its read APIs need to pull audit data from multiple regions and return a unified view. Design-wise, when the aggregation runs and some regions are reachable but others aren't, what's the API's contract? Full failure, partial data with a marker, or keep retrying until complete? And — why that choice?"
"BAM 是一个跨 region 聚合的服务——它的 read API 需要从多个 region 拉 audit 数据并返回统一视图。在设计上，当聚合执行时，部分 region 可达、部分不可达，API 的契约是什么——完全失败 / 部分数据 + 标记 / 一直 retry 到完整？以及——为什么这么选？"

🎯 期望听到的内容 / What you want to hear:

  • 明确选边并讲理由：真做过这种服务的人，几乎一定会选 partial data + 显式 marker（response 里把 unreachable 的 region 列出来），而不是 all-or-nothing。理由：审计用户不能因为 1 个 region 短暂不可达就完全失明 / Picks partial data + explicit marker and explains why: audit users can't go fully blind because one region briefly fails
  • 能讲出 "数据为零" vs "数据缺失"的区分必须做出来 — 否则客户会把 unreachable 当成"该 region 没违规" / Articulates that "zero data" vs "data missing" must be distinguishable — otherwise customers will read unreachable as "no violations in that region"
  • 知道这个契约对客户侧 SDK / dashboard 的下游影响：dashboard 上要专门展示 unreachable 状态、alerts 要把 partial response 区别对待 / Knows the downstream impact on customer SDK / dashboard
  • 能讲 freshness / staleness 处理：partial response 里能不能附带每个 region 的数据时间戳，让客户知道"这部分数据是 5 分钟前的" / Mentions per-region freshness / timestamp in the partial response
  • 🌟 加分：主动提"全部 retry 到完整"这个选项为什么不能选—— audit query 是同步触发的（客户 dashboard 在等），不能为了完整性把延迟做到几分钟 / Bonus: spontaneously explains why "retry until complete" is wrong — audit queries are sync-driven (customer dashboard waiting), can't blow latency to minutes for completeness

🚩 红旗信号 / Red flags:

  • "我们 retry" 一句带过 — 没有 retry 上限、没有最终 fallback 行为 / "We retry" with no upper bound and no final fallback
  • "AWS 自己处理" / "SDK 会自动重试" — 没意识到这是 service team 的 API 契约问题，不是 SDK 问题 / "AWS handles it" / "SDK auto-retries" — doesn't see this is the service team's API contract, not an SDK problem
  • 选 "完全失败"（response 全部 fail） — 在 audit 场景里这是反客户的，1 个 region 抖动让全 org 看不到 audit 状态 / Picks "full failure" — in audit context this is anti-customer; one region's blip blinds the whole org
  • 说不出客户怎么区分 "unreachable" 和 "no data" / Can't explain how customer distinguishes "unreachable" from "no data"

💡 如果她答得太短 / If answer is too short — 压一下细节:

"OK, you return partial data with a marker. Concretely — what does that marker look like in the response payload? And what does the client SDK do with it?"
"好，你返 partial data + marker。具体来说——这个 marker 在 response payload 里长什么样？client SDK 拿到之后会做什么？"

↪ 追问 3 / Follow-up 3 — Rollout 纪律 / Rollout Discipline 🌟🌟 真做过 AWS 的人才答得出

🎤 提问 / Ask
"On the deployment side — you went from N regions to 10. Did you ship all 10 at once or staged it? When region 6 was done and you were about to enable region 7, what was your concrete safety check — what signal said 'green light to proceed'? And if region 7 deploy went sideways, what's the rollback path?"
"部署这边——你从 N 个 region 涨到 10 个，是一次性上 10 个还是分批 stage 的？region 6 上完准备 enable region 7 时，你的具体 safety check 是什么——看到什么 signal 才算"绿灯放行"？如果 region 7 部署翻车，rollback 的路径是什么？"

🎯 期望听到的内容 / What you want to hear:

  • 分批 + wave-based：上一个 region 后有明确的 bake time（24h / 一周）才进下一个 / Staged + wave-based, with explicit bake time per region before progressing
  • 每个 region 内部还要再分阶段：先 internal account / canary fleet → 然后小批客户 → 然后全 region。不是 region 内一把 all-on / Within each region, further staged: internal/canary → small customer cohort → full region. Not all-on at region level
  • 具体 safety signals：error rate 阈值（如 API 4xx/5xx 在 baseline + Xσ 内）/ p99 latency 没退化 / 关键业务 metric（成功 audit 完成率）没掉 / 没有新 high-sev ticket / Concrete safety signals: error rate within baseline + Xσ; p99 latency unchanged; key business metric (audit completion rate) flat; no new high-sev tickets
  • Rollback 路径具体：用 feature flag（per-region toggle）秒级关掉新行为；deploy rollback 用 pipeline 的 previous artifact；数据层如果是 schema 改了，要么 backward-compatible，要么有 expand-contract 双写期 / Concrete rollback: per-region feature flag for sub-second disable; pipeline previous-artifact rollback; schema either backward-compatible or expand-contract dual-write phase
  • 🌟 真 AWS 内部信号：能讲出 cell-based deployment（一个 region 内多 cell，blast radius 隔离）/ AWS 内部 deployment 工具的概念（不一定要说出名字，但能描述"自动化 pipeline 带阶段门禁 + 自动 alarm 触发 rollback"）/ blast radius 这个词主动提 / Real AWS-internal signals: mentions cell-based deployment within a region for blast-radius isolation; describes internal pipeline behavior (auto stage gates + alarm-triggered auto rollback); uses "blast radius" spontaneously

🚩 红旗信号 / Red flags:

  • "一次性上 10 个" — 在 AWS 这等于失业级错误，正经 AWS service 没人这么做 / "All 10 at once" — in AWS this is a fireable mistake; no real AWS service does this
  • "我们 CI/CD 自动部署" — 没回答问题，CI/CD 不是 safety check / "We have CI/CD" — doesn't answer the question; CI/CD is not a safety check
  • safety check 全是定性词（"看起来正常"、"没人投诉"）— 没具体 metric + 阈值 / All-qualitative safety checks ("looks fine", "no one complained") — no specific metric + threshold
  • "出问题 redeploy 旧版本" 但说不出怎么 redeploy、需要多久、新写入的数据怎么办 / "Redeploy old version" but can't explain how, how long, or what happens to data written in the new version
  • 完全没提 feature flag / blast radius / cell / canary 中的任何一个 — AWS deployment 词汇表里这些是基础词 / Mentions none of: feature flag / blast radius / cell / canary — these are baseline AWS deployment vocabulary

💎 加分项 / Bonus signals:

  • 能讲一次真实 rollback 案例（"我们在 region X 部署到 50% 时 alarm 触发，自动 rollback 了，事后定位是 Y"）— 真扛过 oncall 的人才有这种故事 / Tells a real rollback story ("we rolled back at 50% in region X when alarm fired, post-mortem found Y") — only oncall veterans have such stories
  • 主动区分 region cell deployment（同 region 多 cell 隔离）vs cross-region wave（不同 region 之间的顺序）/ Distinguishes intra-region cell deployment vs cross-region wave
  • 提到 data plane vs control plane 是分开 rollout 的（data plane 慢得多）/ Mentions data plane and control plane roll out separately (data plane is far slower)

💡 如果她答得太抽象 / If too abstract — 压具体的:

"Let's get concrete. Pick the moment between region 6 (already live for, say, a week) and starting region 7. What dashboards are you looking at? What's the threshold on each one that flips your decision from 'go' to 'wait'?"
"我们具体一点。挑 region 6 上线一周、准备开 region 7 那一刻——你在看哪些 dashboard？每个 dashboard 上具体什么阈值会让你从"go"翻成"wait"？"
