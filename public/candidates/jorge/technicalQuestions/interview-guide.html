✅ WebSocket 连接管理 / Connection Management

  • 自己封装 vs 三方库：可能用原生 WebSocket API 封装一个 hook，或用 socket.io-client / reconnecting-websocket
    Custom hook wrapping native WebSocket API, or library like socket.io-client / reconnecting-websocket
  • 重连策略：指数退避 (exponential backoff)，最大重试次数，断线 UI 提示
    Exponential backoff, max retries, disconnection UI indicator
  • 心跳机制：定时 ping/pong 检测连接是否存活
    Periodic ping/pong to detect connection liveness

✅ 高频数据处理 / High-frequency Data Handling

  • 节流 (throttle)：每 100-200ms 只取最新值更新 UI，而不是每条消息都触发 render
    Throttle to latest value every 100-200ms instead of rendering every message
  • 批量更新 (batch)：收集一段时间的消息后一次性更新 state，配合 requestAnimationFrame
    Collect messages over a time window, update state once, use requestAnimationFrame
  • 外部 store (Zustand)：在 store 层更新数据，组件只订阅自己需要的 slice，避免无关组件 re-render
    Update in store layer, components subscribe to only their slice, avoids unrelated re-renders
  • 数据层解耦：WebSocket 消息先写入内存数据结构，UI 层按需读取，不直接与 WebSocket 耦合
    WS messages write to in-memory data structure, UI reads on demand, not directly coupled to WS

✅ React 性能优化 / React Performance Optimization

  • React.memo：包装频繁更新的子组件，阻止父组件 render 时的级联更新
    Wrap frequently updating child components to prevent cascade re-renders from parent
  • useMemo / useCallback：缓存计算结果和回调函数，减少引用变化导致的 re-render
    Cache computed results and callbacks to reduce reference-change-triggered re-renders
  • Zustand selector：useStore(state => state.price) 只订阅需要的字段，而不是整个 store
    useStore(state => state.price) subscribes to only needed field, not entire store
  • 组件拆分：把高频更新部分（如价格跳动）拆成独立小组件，不影响其他静态 UI
    Split high-frequency parts (e.g. price ticker) into isolated small components, don't affect static UI
  • 虚拟列表 (virtualization)：订单簿、交易历史等长列表用 react-virtual / react-window
    Use react-virtual / react-window for order book, trade history long lists

📦 Webpack Module Federation

实现方式 / How it works

Webpack 5 原生支持的运行时模块联邦。每个微前端打包为独立构建，通过 ModuleFederationPlugin 在运行时从远程 URL 加载模块。壳应用 (Host) 声明 remotes，子应用 (Remote) 暴露 exposes。
Webpack 5 native runtime module federation. Each MFE is a separate build. The host declares remotes, each remote exposes modules via exposes. Modules are loaded at runtime from remote URLs.

✅ 优势 / Pros — 实际项目中的好处

  • 共享依赖：通过 shared 配置，React / React-DOM / 设计系统等只加载一份，避免 bundle 膨胀
    Shared config ensures React/React-DOM/design system loaded once — avoids bundle bloat
  • 无需额外框架：不像 single-spa 需要引入 orchestrator，MF 就是 Webpack 原生能力
    No extra orchestrator framework needed — it's a native Webpack capability
  • 渐进式采用：可以在现有 Webpack 项目中逐步拆分，不需要一次性重写
    Can gradually split existing Webpack projects without full rewrite
  • 类型安全：搭配 @module-federation/typescript 插件可获得跨应用类型提示
    With @module-federation/typescript plugin, get cross-app type hints

❌ 劣势 / Cons — 实际项目中踩过的坑

  • 强绑定 Webpack 5：如果子应用要迁移到 Vite / Rspack，MF 配置需要大改；Vite 有 vite-plugin-federation 但生态不如原生成熟
    Tightly coupled to Webpack 5 — migrating to Vite/Rspack requires significant config changes
  • 版本对齐问题：如果 Host 和 Remote 的 React 版本不一致，运行时会出现 hook 规则冲突 (Invalid hook call)，排查困难
    React version mismatch between host/remote causes "Invalid hook call" errors — hard to debug
  • 本地开发体验差：需要同时启动多个 dev server，或用 remotes 指向线上地址，调试链路长
    Dev experience suffers — need multiple dev servers running or point remotes to deployed URLs
  • 部署协调：Remote 更新后 Host 不刷新缓存可能加载旧版本，需要配合 CDN 版本号或 content hash 策略
    After remote update, host may cache old version — need CDN versioning or content hash strategy

🎯 适用场景 / Best for

全 React/Webpack 技术栈的大型项目，多团队并行开发，需要运行时集成且不想引入额外框架。
Large all-React/Webpack projects with multiple teams needing runtime integration without extra frameworks.

🔗 single-spa

实现方式 / How it works

框架无关的微前端编排器。由一个 root-config（壳应用）根据 URL 路由注册和激活子应用。每个子应用需要实现 bootstrap, mount, unmount 三个生命周期函数。支持 React / Vue / Angular 等任意框架混合。
Framework-agnostic MFE orchestrator. A root-config registers/activates sub-apps by URL route. Each sub-app implements bootstrap, mount, unmount lifecycle functions. Supports any framework mix.

✅ 优势 / Pros

  • 多框架共存：可以在同一个页面里运行 React 16 + React 18 + Vue 3 的不同模块，适合遗留系统渐进式升级
    Run React 16 + React 18 + Vue 3 on same page — perfect for gradual legacy migration
  • 生态成熟：有 single-spa-react / single-spa-vue 等官方适配器，文档丰富
    Mature ecosystem with official adapters (single-spa-react/vue), rich documentation
  • 构建工具无关：子应用用 Webpack / Vite / Rollup 都行，不强制统一
    Build-tool agnostic — sub-apps can use Webpack/Vite/Rollup independently

❌ 劣势 / Cons

  • 配置复杂：root-config + importmap + SystemJS 的配置链很长，新人上手至少 1-2 天才能理清
    Complex config chain (root-config + importmap + SystemJS) — new devs need 1-2 days to understand
  • 样式泄漏：不自带样式隔离，子应用的全局 CSS 会互相污染，需要手动用 CSS Modules / Shadow DOM 处理
    No built-in style isolation — sub-app global CSS leaks to others, need manual CSS Modules or Shadow DOM
  • 共享依赖困难：不像 MF 有原生 shared 机制，需要自己通过 importmap 或 externals 管理，容易出错
    No native shared mechanism like MF — must manage via importmap/externals, error-prone
  • 生命周期 bug：子应用 mount/unmount 如果有副作用清理不干净（事件监听、定时器），会导致内存泄漏和幽灵 bug
    If sub-app mount/unmount has unclean side effects (event listeners, timers), causes memory leaks and ghost bugs

🎯 适用场景 / Best for

多框架混合的大型项目，尤其是需要从 jQuery/Angular/Vue 渐进式迁移到 React 的场景。
Large multi-framework projects, especially gradual migration from jQuery/Angular/Vue to React.

🪟 iframe 隔离 / iframe Isolation

实现方式 / How it works

每个微前端在独立的 <iframe> 中运行，壳应用通过 postMessage API 与子应用通信。每个 iframe 有自己的 JS 执行上下文、DOM 树和 CSS 作用域。
Each MFE runs in its own <iframe>. The shell communicates via postMessage. Each iframe has isolated JS context, DOM tree, and CSS scope.

✅ 优势 / Pros

  • 完全隔离：JS / CSS / DOM 完全沙箱化，子应用之间无论怎么写都不会互相影响
    Complete sandboxing — JS/CSS/DOM fully isolated, sub-apps can't affect each other no matter what
  • 安全性最高：适合嵌入第三方不可信代码（如广告、合作伙伴模块）
    Highest security — suitable for embedding untrusted third-party code (ads, partner modules)
  • 零改造：已有独立应用可以直接嵌入 iframe，不需要任何代码改造
    Zero refactoring — existing standalone apps can be embedded directly

❌ 劣势 / Cons

  • 性能差：每个 iframe 加载一套完整 HTML/CSS/JS，内存消耗 2-5 倍，首次加载明显卡顿
    Each iframe loads full HTML/CSS/JS — 2-5x memory, noticeably slow first load
  • 通信困难：只能用 postMessage，序列化/反序列化开销大，无法共享 React context 或 store
    Only postMessage available — serialization overhead, can't share React context or store
  • UX 割裂：跨 iframe 的弹窗无法覆盖父页面、滚动条独立、键盘焦点容易丢失
    Modals can't overlay parent page, separate scrollbars, keyboard focus easily lost
  • SEO 不友好：搜索引擎无法爬取 iframe 内容
    Search engines can't crawl iframe content

🎯 适用场景 / Best for

安全隔离要求极高的场景（金融合规、第三方嵌入），或遗留系统快速集成且不需要频繁交互的情况。
High-security isolation needs (financial compliance, third-party embedding), or quick legacy integration with minimal interaction.

🔀 路由分发 / Route-based (Nginx)

实现方式 / How it works

通过反向代理（通常是 Nginx）按 URL 路径将请求分发到不同的独立应用。例如 /trading/* 代理到交易应用，/admin/* 代理到管理后台。每个应用是完全独立的 SPA，有自己的 index.html。
Reverse proxy (usually Nginx) routes requests to different standalone apps by URL path. E.g., /trading/* proxies to trading app, /admin/* to admin panel. Each app is a fully independent SPA with its own index.html.

✅ 优势 / Pros

  • 极简：纯运维配置，不需要任何前端框架层面的改造，Nginx 配几行就完成
    Ultra-simple — pure infra config, no frontend framework changes needed, a few Nginx lines
  • 完全独立：每个应用可以用任意框架、任意版本、任意构建工具，零耦合
    Fully independent — each app can use any framework, any version, any build tool, zero coupling
  • 部署最简单：每个应用独立部署到不同的路径/域名，CI/CD 完全解耦
    Simplest deployment — each app deploys independently to different path/domain, fully decoupled CI/CD

❌ 劣势 / Cons

  • 整页刷新：从 /trading 跳到 /admin 时浏览器整页刷新，用户体验差，状态全部丢失
    Full-page refresh when navigating between apps — bad UX, all state lost
  • 无法共享状态：不同应用之间只能通过 URL 参数、cookies 或后端 session 共享数据
    No shared state — apps can only share data via URL params, cookies, or backend sessions
  • UI 不一致风险：每个应用独立维护 UI 组件，设计系统容易跑偏
    UI inconsistency risk — each app maintains its own UI components, design system easily drifts
  • 重复加载：公共依赖（React、UI 库）每个应用都打包一份，用户每次跳转都重新下载
    Duplicate loading — shared deps bundled separately per app, user re-downloads on every navigation

🎯 适用场景 / Best for

各模块交互很少的内部平台（如后台管理系统的不同模块），或团队规模小、不值得引入复杂微前端框架的场景。
Internal platforms with minimal cross-module interaction (e.g., admin dashboards), or small teams where complex MFE frameworks aren't worth the overhead.
