Flow 1 · 🏗️ 系统架构 / System Architecture ~10 min

📌 简历原文 / Resume excerpt — AWS Vancouver · Senior Software Engineer (Jan 2024 – Feb 2025):

"Led the design and implementation of high-level system architecture for distributed applications using C++ and Java, ensuring robust backend scalability."

"主导了分布式应用的高层系统架构设计与实现，使用 C++ 和 Java，确保了稳健的后端可扩展性。"

🎯 考察意图 / Probing intent：这条 bullet 充满了 buzzword（Led / high-level architecture / distributed / scalability）但没有任何具体名词。目标是逐层戳穿，验证他在 Senior 级别是否真的主导了架构设计，还是只是参与者。
This bullet is full of buzzwords (Led / high-level architecture / distributed / scalability) with no concrete specifics. The goal is to probe layer by layer and verify whether he genuinely led the architecture work at Senior level, or was just a participant.

🎤 主问题 / Main Question — 项目背景 / Project Background

🎤 提问 / Ask
"Looking at your most recent role at AWS in Vancouver, the first bullet says you led the design and implementation of a high-level system architecture for a distributed application using C++ and Java. Could you start by walking me through the background of this project? What was the product or service, what problem was it solving, and what made this work necessary?"
"看你最近在 AWS 温哥华的这段经历，第一条 bullet 说你主导了一个用 C++ 和 Java 构建的分布式应用的高层系统架构设计和实现。能先给我介绍一下这个项目的背景吗？这是什么产品或服务？要解决什么问题？为什么需要做这个工作？"

🎯 期望听到的内容 / What you want to hear:

  • 具体的服务名 / product name（哪个 AWS 服务？internal 还是 customer-facing？）/ A specific service or product name
  • 明确的业务问题——"为什么需要重新设计架构" / A clear business / engineering problem motivating the work
  • 项目的规模和约束（QPS、数据量、SLA、团队规模）/ Scope & constraints (QPS, data volume, SLA, team size)
  • 他自己的角色——"Led" 到底是 DRI、tech lead、还是 IC 之一 / His actual role — DRI vs tech lead vs one IC among many

🚩 红旗信号 / Red flags:

  • 说不出具体服务名 / 只能讲 "a distributed system" / Can't name the service — only "a distributed system"
  • 讲不清楚 business problem，全是技术 buzzword / Vague on the business problem, only technical buzzwords
  • "Led" 但说不清队伍人数 / 自己的具体决策权 / "Led" but can't articulate team size or his decision authority
  • 全程 "we did X"，从来不说 "I" / Always "we did X", never "I"

↪ 追问 1 / Follow-up 1 — 架构具体是什么 / Architecture Specifics

🎤 提问 / Ask
"Now let's go deeper on the architecture itself. You said you designed the high-level architecture — what exactly did you design? Walk me through the major components, how they communicate, and which parts are in C++ vs Java. Feel free to sketch it out if it helps."
"现在我们具体聊聊架构本身。你说你设计了高层架构——具体设计了什么？给我介绍一下主要的组件、它们之间怎么通信、哪些部分用 C++、哪些部分用 Java。如果方便的话可以画一下。"

🎯 期望听到的内容 / What you want to hear:

  • 60 秒内能"画"出 5+ 个组件 + 箭头 + 数据流向——真正主导过架构的人这一步不会卡 / Can sketch 5+ components + arrows + data flow within 60 seconds
  • C++ 和 Java 的分工有清晰理由（如：C++ 写延迟敏感的数据面 data plane，Java 写控制面 control plane / 业务逻辑）/ Clear rationale for the C++/Java split (e.g., C++ for latency-critical data plane, Java for control plane / business logic)
  • 明确的组件间通信方式（gRPC? Kafka? 共享 protobuf? JNI?）/ Explicit inter-component communication (gRPC, Kafka, shared protobuf, JNI)
  • 能讲出 distributed 维度：分区策略 / 一致性模型 / failure handling / Distributed dimension: partitioning, consistency model, failure handling

🚩 红旗信号 / Red flags:

  • 只能讲 "we have a frontend, a backend, a database..."，给不出具体组件名 / Generic "frontend, backend, database" without named components
  • C++ 和 Java 的分工说不清楚理由（"都是历史原因 / 团队习惯"）/ Can't justify the C++/Java split with engineering reasoning
  • "distributed" 但只能描述一个 stateless 服务 + load balancer，没有多节点协调的细节 / "Distributed" but only describes one stateless service + LB, no multi-node coordination
  • 说不出任何具体的 design pattern 名词（leader/follower、sharding、Paxos/Raft、gossip 等）/ Can't name any specific architecture patterns

💡 如果他卡壳 / If stuck — 给一个起点 / Anchor the question:

"For example — was this a single service split into multiple processes, or multiple cooperating services? And inside one service, was there a clear data plane / control plane separation?"
"比如说——这是一个单服务拆成多个进程，还是多个协作的服务？在单个服务内部，有没有明确的 data plane / control plane 划分？"

📝 Flow 1 还可加：design decision debate / Senior-bar 陷阱题等。/ Flow 1 can be extended with design-decision debates and Senior-bar traps next.

Flow 2 · ⚡ 性能优化 / Performance Optimization ~8 min

📌 简历原文 / Resume excerpt — AWS Vancouver · Senior Software Engineer (Jan 2024 – Feb 2025):

"Optimized performance of complex distributed systems, significantly enhancing throughput and reducing latency in Agile development environments."

"在敏捷开发环境中优化了复杂分布式系统的性能，显著提升了吞吐量并降低了延迟。"

🎯 考察意图 / Probing intent：整段简历里 buzzword 密度最高的一条——"Optimized / complex / significantly / throughput / latency / Agile" 每个词都是填充，没有一个具体名词、一个数字。真做过 perf 优化的 SWE 简历上一定会留数字。目标是用「先要数字 → 再问做法 → 最后问 trade-off」三步把虚的部分戳穿。
The most buzzword-dense bullet in the resume — "Optimized / complex / significantly / throughput / latency / Agile" are all filler, with no concrete names or numbers. Real perf engineers always leave numbers on their resume. Drill via: get numbers → get the actual change → get the trade-off.

🔗 与 Flow 1 的衔接 / Bridge from Flow 1：这条 bullet 很可能就是 Flow 1 同一个项目的延续——可以自然地承接 "Now still on this same system you described, the next bullet says you optimized its performance..."。/ This bullet likely refers to the same project from Flow 1 — bridge naturally.

🎤 主问题 / Main Question — 哪个系统 + 项目详情 / Which System + Project Details

🎤 提问 / Ask
"The next bullet says you optimized performance of complex distributed systems. Which system specifically? Could you walk me through the detailed context of the project this system corresponds to — what it does, who uses it, and what was happening with its performance that made optimization necessary?"
"下一条 bullet 说你优化了复杂分布式系统的性能。具体是哪个系统？可以介绍一下这个 system 对应的项目详细情况吗？它是干什么的？谁在用？当时性能上发生了什么，让你必须去优化？"

🎯 期望听到的内容 / What you want to hear:

  • 30 秒内说出具体系统名 / 模块名（不是 "a service" / "the platform"）/ Specific system / module name within 30s
  • 明确的触发事件——为什么这时候要优化？（SLA 即将违反？客户投诉？容量规划撞墙？新业务上量？）/ Clear trigger — why optimize now?
  • 用户 / 调用方画像（internal AWS team? customer-facing? 多少 QPS?）/ Caller profile (internal team, customer-facing, QPS)
  • 自己在这个优化里的角色（一个人扛？带 N 人？还是某个 IC 的一部分工作？）/ His personal role in the optimization effort

🚩 红旗信号 / Red flags:

  • 说不出具体系统名，或者跟 Flow 1 是"完全不同的另一个系统"但也讲不清楚 / Can't name the system, or claims a different system but can't describe it
  • 讲不清楚为什么需要这次优化 — 全是 "to improve performance" 的循环论证 / Circular reasoning ("to improve perf")
  • "我们整体的吞吐"——根本没具体场景 / "Our overall throughput" with no concrete scenario

↪ 追问 1 / Follow-up 1 — 要数字 / Show Me the Numbers

🎤 提问 / Ask
"Can you give me the numbers? What was throughput / latency before, what did it become after, and over what timeframe?"
"能给我具体数字吗？优化前的 throughput / latency 是多少？优化后变成多少？整个周期持续了多长时间？"

🎯 期望听到的内容 / What you want to hear:

  • 至少一对 before / after 数字（单位明确：ms / QPS / RPS / P50 / P99）/ At least one concrete before / after pair (with units)
  • 清晰的 时间线（"我从 X 月开始 owner 这块，到 Y 月达到目标"）/ Clear timeline
  • 能解释数字的测量方式——synthetic benchmark? production traffic shadowing? canary? / How was the number measured?
  • 记得自己的目标是什么（SLA 数字？容量目标？）/ Remembers the actual goal

🚩 红旗信号（这是 Flow 2 最致命的一关）/ Red flags (the most lethal gate in Flow 2):

  • ❗ "I don't remember exact numbers" — Senior 级别记不住自己最近一年最核心成就的数字 = 没做过 / Senior who can't recall key achievement numbers from the last year = didn't do it
  • ❗ 只复述简历原话 "throughput improved a lot / latency went down significantly" / Just re-reads the resume
  • 给出过于"圆满"的行业 generic 数字（"P99 减半"、"throughput 2x"——所有人都说这个）/ Suspiciously round industry-generic numbers
  • 说不出测量方式 / 把 dev box benchmark 当 production 数字 / Confuses dev-box bench with production numbers

💡 如果他卡壳 / If stuck — 降低门槛 / Lower the bar:

"Even a rough estimate is fine — were we talking about a 10% improvement or 10x? Was the baseline 10ms or 1s?"
"粗略的估计也行——是 10% 的提升还是 10 倍？基线是 10ms 还是 1s 那个量级？"

注：如果连量级都给不出，那基本可以判定这条 bullet 是包装/凑数。/ Note: if he can't even give an order of magnitude, the bullet is almost certainly padded.

↪ 追问 2 / Follow-up 2 — 怎么做的 + 为什么这种方案 / How & Why This Approach

🎤 提问 / Ask
"Walk me through what you actually did to optimize it. And — why did you choose this approach over alternatives? What other options did you consider, and what were the trade-offs?"
"给我讲讲你具体做了什么优化？以及——为什么选择了这种方案，而不是别的？当时还考虑过哪些其他方案？做了哪些权衡？"

🎯 期望听到的内容 / What you want to hear:

  • 具体的改动（不是 "we added caching"，而是 "we added a write-through Redis cache with 5min TTL on the X path, partitioned by Y"）/ Concrete change with details, not generic "we added caching"
  • 诊断 → 决策的因果链：先看到 X 数据 → 怀疑是 Y → 验证后定位到 Z → 才决定做 W / Causal chain from diagnosis to decision
  • 至少考虑过 2 个备选方案，能解释为什么没选 / At least 2 alternatives considered, with rationale for rejection
  • 能讲出明确的 trade-off：内存换 CPU / 一致性换 latency / 复杂度换性能 / Explicit trade-offs (mem ↔ CPU, consistency ↔ latency, complexity ↔ perf)
  • 知道用了什么工具诊断（profiler 名字、flame graph、distributed tracing、metrics 系统名）/ Names the tools used for diagnosis

🚩 红旗信号 / Red flags:

  • 跳过诊断过程，直接说 "we added caching / batching / async" — 没诊断 = 跟着别人做的 / Skips diagnosis, jumps to solution = followed someone else
  • "didn't consider alternatives, this was obviously the right approach" — 资深工程师必有备选 / No alternatives considered = not a senior-level decision
  • 说不出任何 trade-off ("没有什么 downside") — 任何优化都有代价 / Claims no downside
  • 工具链全是开源词（Grafana、Prometheus）— AWS 内部基本不用这些 / All open-source tools (Grafana/Prometheus) — AWS internal mostly doesn't use these
  • "everything went smoothly, no rollback" — 太完美就是没做过 / Too smooth = didn't do it

💡 可选追加 / Optional add-on — 交叉验证陷阱 / Cross-check trap:

"Earlier on your resume, there's a bullet about 'Read Singleton Time' from your Seattle role — that also sounds like latency work. How do these two compare? Was the optimization approach similar or different?"
"你简历早期的 Seattle 那段有一条提到 'Read Singleton Time'——那也像是 latency 相关的工作。这两个对比一下，优化思路是相似还是不同？"

用途：Seattle 那条是整份简历里唯一带具体技术名词的 perf 内容，可用于交叉验证他对 perf 工作的真实理解深度。/ Use: the Seattle bullet is the only perf content with concrete technical names — useful to cross-verify his actual depth.
