✅ WebSocket 连接管理 / Connection Management

  • 自己封装 vs 三方库：可能用原生 WebSocket API 封装一个 hook，或用 socket.io-client / reconnecting-websocket
    Custom hook wrapping native WebSocket API, or library like socket.io-client / reconnecting-websocket
  • 重连策略：指数退避 (exponential backoff)，最大重试次数，断线 UI 提示
    Exponential backoff, max retries, disconnection UI indicator
  • 心跳机制：定时 ping/pong 检测连接是否存活
    Periodic ping/pong to detect connection liveness

✅ 高频数据处理 / High-frequency Data Handling

  • 节流 (throttle)：每 100-200ms 只取最新值更新 UI，而不是每条消息都触发 render
    Throttle to latest value every 100-200ms instead of rendering every message
  • 批量更新 (batch)：收集一段时间的消息后一次性更新 state，配合 requestAnimationFrame
    Collect messages over a time window, update state once, use requestAnimationFrame
  • 外部 store (Zustand)：在 store 层更新数据，组件只订阅自己需要的 slice，避免无关组件 re-render
    Update in store layer, components subscribe to only their slice, avoids unrelated re-renders
  • 数据层解耦：WebSocket 消息先写入内存数据结构，UI 层按需读取，不直接与 WebSocket 耦合
    WS messages write to in-memory data structure, UI reads on demand, not directly coupled to WS

✅ React 性能优化 / React Performance Optimization

  • React.memo：包装频繁更新的子组件，阻止父组件 render 时的级联更新
    Wrap frequently updating child components to prevent cascade re-renders from parent
  • useMemo / useCallback：缓存计算结果和回调函数，减少引用变化导致的 re-render
    Cache computed results and callbacks to reduce reference-change-triggered re-renders
  • Zustand selector：useStore(state => state.price) 只订阅需要的字段，而不是整个 store
    useStore(state => state.price) subscribes to only needed field, not entire store
  • 组件拆分：把高频更新部分（如价格跳动）拆成独立小组件，不影响其他静态 UI
    Split high-frequency parts (e.g. price ticker) into isolated small components, don't affect static UI
  • 虚拟列表 (virtualization)：订单簿、交易历史等长列表用 react-virtual / react-window
    Use react-virtual / react-window for order book, trade history long lists

Q3.1 项目背景与选型动机 / Why Micro-Frontends

合理的动机 / Reasonable motivations:

  • 🟢 多个团队并行开发不同业务模块，单体应用合并冲突太多 — Multiple teams on different modules, too many merge conflicts in monolith
  • 🟢 各模块需要独立发布节奏，不能因为一个模块的 bug 阻塞整个产品 — Independent release cadence, one bug shouldn't block entire product
  • 🟢 交易所有多个不同的应用上下文（如交易大厅、订单管理、数据报表） — Exchange has distinct contexts (trading floor, order management, reporting)

Q3.2 技术方案选型 / Technical Approach

根据简历上下文（react-redux-saga + Webpack），最可能的方案：
Based on resume context (react-redux-saga + Webpack), most likely approaches:

  • 🟢 Webpack Module Federation — 最合理，全 React 生态 + Webpack 构建
    Most likely — all-React ecosystem + Webpack builds
  • 🟡 single-spa — 也合理，尤其有遗留代码需渐进式集成
    Plausible if legacy code needed gradual integration
  • 🟡 路由分发 (Nginx) — 可能，如果各模块交互不多
    Possible if modules had minimal interaction

Q3.3 拆分方式与通信 / Splitting & Communication

拆分方式 / Splitting approach:

  • 🟢 按业务域：交易模块、订单管理、用户设置、报表 — By business domain: trading, order mgmt, user settings, reports
  • 🟢 按团队：每个团队拥有一个微前端 — By team: each team owns one MFE
  • 🟡 按页面 — By page (coarser, but feasible)

通信方式 / Communication:

  • 🟢 Custom Events (window.dispatchEvent) — 松耦合 / Loosely-coupled
  • 🟢 共享 store (Zustand/Redux via Module Federation) — Shared store instance
  • 🟡 Event Bus / PubSub — Custom lightweight pub/sub
  • 🟡 URL / 路由参数 — URL query params (limited)
  • 🟠 postMessage (iframe) — Required for iframe approach
